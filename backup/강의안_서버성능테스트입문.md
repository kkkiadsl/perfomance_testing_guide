## 1. 성능 테스트가 뭔가요? (개요)

### 1.1 성능 테스트란?

성능 테스트는 **서버가 얼마나 많은 사용자의 요청을 안정적으로 처리할 수 있는지** 측정하는 테스트입니다.

우리가 평소에 하는 기능 테스트는 "이 기능이 올바르게 동작하는가?"를 확인합니다.
성능 테스트는 여기서 한 걸음 더 나아가 **"수백, 수천 명이 동시에 사용해도 올바르게 동작하는가?"** 를 확인합니다.

> **비유로 이해하기**
>
> 엘리베이터를 생각해 보세요.
> - 기능 테스트: 버튼을 누르면 해당 층으로 이동하는가?
> - 성능 테스트: 정원 10명인 엘리베이터에 15명이 타면 어떻게 되는가?

### 1.2 왜 QA 엔지니어가 성능 테스트를 알아야 할까?

| 이유 | 설명 |
|------|------|
| **품질의 완성** | 기능이 정상이어도 느리면 사용자는 떠납니다 |
| **장애 예방** | 출시 전에 서버의 한계를 파악해야 장애를 예방할 수 있습니다 |
| **근거 기반 판단** | "느린 것 같아요"가 아닌, 수치로 품질을 이야기할 수 있습니다 |
| **커뮤니케이션** | 개발팀과 성능 이슈를 정량적으로 논의할 수 있습니다 |

### 1.3 성능 테스트의 종류

```
부하 강도 →
──────────────────────────────────────────────►

┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│  부하    │   │ 스트레스 │   │  스파이크 │   │  내구성  │
│ (Load)  │   │(Stress) │   │ (Spike) │   │ (Soak)  │
│         │   │         │   │         │   │         │
│ 예상     │   │ 한계     │   │ 급격한   │   │ 장시간   │
│ 트래픽   │   │ 초과     │   │ 증가     │   │ 지속     │
└─────────┘   └─────────┘   └─────────┘   └─────────┘
```

| 종류 | 목적 | 핵심 질문 |
|------|------|----------|
| **부하 테스트 (Load Test)** | 예상 트래픽에서 정상 동작하는지 확인 | "평소 트래픽을 잘 처리하는가?" |
| **스트레스 테스트 (Stress Test)** | 시스템의 한계점(Breaking Point)을 파악 | "어디까지 버틸 수 있는가?" |
| **스파이크 테스트 (Spike Test)** | 갑작스러운 트래픽 급증에 대한 대응력 확인 | "갑자기 몰리면 어떻게 되는가?" |
| **내구성 테스트 (Soak Test)** | 장시간 운영 시 메모리 누수 등 확인 | "오래 돌려도 괜찮은가?" |

## 2. 자세히 알아보기 (상세 이론)

### 2.1 핵심 성능 지표 (Metrics)

성능 테스트의 결과를 읽으려면, 먼저 **핵심 지표**를 이해해야 합니다.

#### 2.1.1 응답 시간 (Response Time)

서버가 요청을 받고 응답을 돌려주기까지 걸리는 시간입니다.

```
클라이언트                    서버
    │                         │
    │──── 요청 (Request) ────►│
    │                         │ ← 처리 중...
    │◄─── 응답 (Response) ────│
    │                         │
    │◄──── 응답 시간 (ms) ────►│
```

- **평균 응답 시간 (Average)**: 전체 응답 시간의 평균
- **중앙값 (Median, p50)**: 전체 요청 중 50%가 이 시간 이내에 완료
- **p90 / p95 / p99**: 전체 요청 중 90% / 95% / 99%가 이 시간 이내에 완료

> **왜 평균만으로는 부족한가?**
>
> | 요청 | 응답 시간 |
> |------|----------|
> | 1~9번 | 각 100ms |
> | 10번 | 10,000ms |
> | **평균** | **1,090ms** |
> | **p90** | **100ms** |
> | **p99** | **10,000ms** |
>
> 평균은 1,090ms로 나쁘지 않아 보이지만, 실제로 10번째 사용자는 10초나 기다렸습니다.
> **p99는 "100명 중 가장 불행한 1명의 경험"** 을 보여줍니다.
> 서비스 품질은 가장 느린 사용자의 경험이 결정합니다.

#### 2.1.2 처리량 (Throughput)

단위 시간당 서버가 처리한 요청 수입니다.

- **RPS (Requests Per Second)**: 초당 처리 요청 수
- **TPS (Transactions Per Second)**: 초당 처리 트랜잭션 수

```
RPS = 총 요청 수 / 총 소요 시간(초)
```

> RPS와 TPS의 차이:
> - RPS: 개별 HTTP 요청 1건 = 1 Request
> - TPS: 비즈니스 단위의 트랜잭션 1건 = 1 Transaction (예: 로그인 → 상품조회 → 결제 = 1 Transaction)

#### 2.1.3 동시 사용자 수 (Concurrent Users)

같은 시간대에 서버에 요청을 보내고 있는 사용자 수입니다.

```
시간 ──►
User A: ████░░░░████░░░░
User B: ░░████░░░░████░░
User C: ░░░░████░░░░████
         ▲
         │ 이 시점의 동시 사용자 = 2명 (A, B)
```

#### 2.1.4 에러율 (Error Rate)

전체 요청 중 실패한 요청의 비율입니다.

```
에러율(%) = (실패 요청 수 / 전체 요청 수) × 100
```

- 일반적으로 **에러율 1% 이하**를 안정적이라고 판단합니다.
- 에러율이 급증하는 지점 = 시스템의 한계점입니다.

### 2.2 성능 테스트의 주요 개념

#### 2.2.1 Ramp-up / Steady / Ramp-down

성능 테스트는 일반적으로 3단계로 구성됩니다.

```
사용자 수
    ▲
    │          ┌──────────────────┐
    │         /│    Steady State   │\
    │        / │  (최대 부하 유지)  │ \
    │       /  │                  │  \
    │      /   │                  │   \
    │     /    │                  │    \
    │    /     │                  │     \
    ├───/──────┴──────────────────┴──────\───► 시간
    │ Ramp-up                      Ramp-down
    │ (점진적 증가)                  (점진적 감소)
```

| 단계 | 설명 | 목적 |
|------|------|------|
| **Ramp-up** | 사용자를 점진적으로 증가시키는 구간 | 갑작스러운 부하로 인한 왜곡 방지 |
| **Steady State** | 목표 사용자 수를 유지하는 구간 | 안정적인 성능 데이터 수집 |
| **Ramp-down** | 사용자를 점진적으로 줄이는 구간 | 부하 해소 후 시스템 복구 확인 |

#### 2.2.2 Think Time (대기 시간)

실제 사용자는 페이지를 읽거나 입력하는 데 시간을 소비합니다. 이를 시뮬레이션하기 위해 요청 사이에 의도적으로 대기 시간을 넣습니다.

```
사용자 행동: [로그인] ──3초 대기── [상품목록 조회] ──5초 대기── [상품 상세 조회]
                         ▲                           ▲
                    Think Time                   Think Time
```

- Think Time이 없으면: 비현실적으로 높은 부하가 발생합니다.
- Think Time이 있으면: 실제 사용자 패턴에 가까운 부하를 생성합니다.

#### 2.2.3 가상 사용자 (Virtual User, VU)

성능 테스트 도구가 생성하는 가상의 사용자입니다. 각 VU는 독립적으로 시나리오를 반복 수행합니다.

```
VU 1: [로그인] → [조회] → [로그인] → [조회] → ...  (반복)
VU 2: [로그인] → [조회] → [로그인] → [조회] → ...  (반복)
VU 3: [로그인] → [조회] → [로그인] → [조회] → ...  (반복)
```

### 2.3 성능 테스트 프로세스

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 1. 목표   │    │ 2. 시나리오│    │ 3. 환경   │    │ 4. 실행   │    │ 5. 분석   │
│    설정   │───►│    설계   │───►│    구성   │───►│  & 모니터링│───►│  & 보고   │
│          │    │          │    │          │    │          │    │          │
│ - 대상 API│    │ - 유저 흐름│    │ - 테스트  │    │ - 테스트  │    │ - 결과    │
│ - 목표 RPS│    │ - 데이터  │    │   서버    │    │   실행    │    │   해석    │
│ - 성공기준│    │ - Think  │    │ - 모니터링│    │ - 실시간  │    │ - 병목    │
│          │    │   Time   │    │   도구    │    │   관찰    │    │   식별    │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

### 2.4 성능 테스트 시 흔한 실수

| 실수 | 왜 문제인가 | 올바른 접근 |
|------|-----------|-----------|
| Think Time 없이 테스트 | 실제보다 훨씬 높은 부하, 비현실적 결과 | 실제 사용자 패턴을 반영한 Think Time 설정 |
| 단일 API만 테스트 | 실제 사용 패턴과 다름 | 사용자 시나리오 기반으로 여러 API 조합 |
| 로컬 환경에서 테스트 | 네트워크, 하드웨어 차이로 의미 없는 결과 | 운영 환경과 유사한 환경에서 실행 |
| Ramp-up 없이 바로 최대 부하 | 서버가 준비 없이 충격을 받아 왜곡된 결과 | 점진적으로 부하를 증가시킴 |
| 1회만 실행하고 결론 | 네트워크 상태 등 변수가 많아 신뢰도 낮음 | 최소 3회 이상 반복 후 평균으로 판단 |

## 3. 어떤 일이 있었나? (실제 사례)

### 사례 1: 티켓 예매 사이트 — "새벽에 열었는데 서버가 터졌어요"

**상황**
- 인기 콘서트 티켓 오픈 시간에 수십만 명이 동시 접속
- 서버가 감당할 수 없는 트래픽이 순간적으로 몰림

**증상**
- 페이지 로딩 시간 30초 이상
- "504 Gateway Timeout" 에러 대량 발생
- 결제 도중 세션 만료, 중복 결제 발생

**성능 테스트 관점에서의 원인**
- **스파이크 테스트 미실시**: 순간 트래픽 급증 시나리오를 사전에 검증하지 않음
- **대기열(Queue) 시스템 부재**: 동시 접속을 제어하는 장치가 없었음

**교훈**
> 이벤트성 서비스는 반드시 **스파이크 테스트**를 수행해야 하며, 예상 최대 동시 접속자의 **2~3배** 수준으로 테스트해야 합니다.

---

### 사례 2: 쇼핑몰 — "블랙프라이데이에 결제가 안 돼요"

**상황**
- 대형 할인 행사 기간에 평소 대비 10배의 트래픽 발생
- 상품 목록 조회는 되지만 결제 API에서 병목 발생

**증상**
- 상품 조회 응답 시간: 200ms (정상)
- 결제 API 응답 시간: 15,000ms → Timeout
- 에러율 45%까지 상승

**성능 테스트 관점에서의 원인**
- **부하 테스트 시 결제 API를 제외**: "결제는 외부 PG사 영역"이라고 판단하여 테스트 범위에서 제외
- **API별 처리량 차이 미파악**: 조회 API와 결제 API의 성능 한계가 다름을 인지하지 못함

**교훈**
> 성능 테스트는 **사용자 시나리오의 전체 흐름**을 대상으로 해야 합니다. 특정 API를 제외하면 가장 약한 고리를 놓치게 됩니다.

---

### 사례 3: 사내 시스템 — "월요일 아침마다 느려요"

**상황**
- 월요일 오전 9시, 전 직원이 동시에 출근 후 사내 시스템 접속
- 일주일 치 알림과 데이터가 한꺼번에 로딩

**증상**
- 대시보드 로딩 시간 평소 2초 → 월요일 15초
- DB 커넥션 풀 고갈로 일부 사용자 접속 불가
- 화요일부터는 정상 복구

**성능 테스트 관점에서의 원인**
- **부하 테스트 시 동시 접속 패턴 미반영**: 특정 시간대 집중 접속 패턴을 고려하지 않음
- **DB 커넥션 풀 크기 검증 부재**: 동시 요청 수 대비 커넥션 풀 설정이 부족

**교훈**
> 성능 테스트는 **시간대별 사용 패턴**을 반영해야 합니다. "평균 사용자 수"가 아닌 "**피크 시간대의 동시 사용자 수**"를 기준으로 설계해야 합니다.

---

### 사례에서 배우는 공통 교훈

```
┌─────────────────────────────────────────────────────────┐
│                    성능 테스트 핵심 원칙                    │
│                                                         │
│  1. 실제 사용 패턴을 반영한 시나리오를 설계하라              │
│  2. 예상 트래픽의 2~3배 수준으로 테스트하라                  │
│  3. 전체 사용자 흐름(End-to-End)을 테스트 범위에 포함하라    │
│  4. 피크 시간대, 이벤트 등 특수 상황을 반드시 고려하라        │
│  5. 결과를 수치로 기록하고, 기준(Baseline)을 만들어라        │
└─────────────────────────────────────────────────────────┘
```

## 4. 실습 전에 확인하기 (환경 점검)

### 4.1 실습 아키텍처

이번 실습에서는 단순히 도구를 실행하는 것을 넘어, **현대적인 성능 모니터링 체계**를 직접 구성합니다.

```
┌─── Locust 흐름 (자체 Web UI) ──────────┐  ┌─── k6 흐름 (Grafana 연동) ────────────┐
│                                         │  │                                       │
│  ┌──────────┐      ┌──────────────┐    │  │  ┌────────┐      ┌──────────────┐    │
│  │  Locust  │─────►│   대상 서버   │    │  │  │   k6   │─────►│   대상 서버   │    │
│  │ (Python) │      │   (Docker)   │    │  │  │  (JS)  │      │   (Docker)   │    │
│  └────┬─────┘      └──────────────┘    │  │  └───┬────┘      └──────────────┘    │
│       │                                │  │      │                                │
│       ▼                                │  │      ▼                                │
│  ┌──────────────────┐                  │  │  ┌──────────┐    ┌───────────────┐    │
```

## 5. 성능 테스트 해보기 (Locust)

### 5.1 Locust란?

Locust는 **Python 기반의 오픈소스 성능 테스트 도구**입니다.

| 항목 | 내용 |
|------|------|
| **언어** | Python |
| **특징** | 코드 기반 시나리오 작성, Web UI 제공 |
| **장점** | Python을 아는 사람이라면 빠르게 시작 가능 |
| **공식 사이트** | locust.io |

### 5.2 Locust의 핵심 구조

```python
from locust import HttpUser, task, between

class MyUser(HttpUser):
    # Think Time: 1~3초 사이 랜덤 대기
    wait_time = between(1, 3)

    @task
    def my_task(self):
        self.client.get("/api/endpoint")
```

| 구성 요소 | 설명 |
|----------|------|
| `HttpUser` | 가상 사용자(VU)를 정의하는 클래스 |
| `wait_time` | 요청 간 대기 시간 (Think Time) |
| `@task` | 가상 사용자가 반복 수행할 작업 |
| `self.client` | HTTP 요청을 보내는 클라이언트 |

### 5.3 실습: 기본 부하 테스트

#### Step 1 — 테스트 스크립트 작성

`locustfile.py` 파일을 생성합니다.

```python
from locust import HttpUser, task, between

class BasicUser(HttpUser):
    """기본 부하 테스트 시나리오"""

    # 요청 간 대기 시간: 1~3초 (실제 사용자 패턴 시뮬레이션)
    wait_time = between(1, 3)

    @task(3)  # 가중치 3: 더 자주 실행됨
    def get_homepage(self):
        """메인 페이지 조회"""
        self.client.get("/", name="메인페이지")

    @task(1)  # 가중치 1: 상대적으로 적게 실행됨
    def get_api(self):
        """API 엔드포인트 조회"""
        self.client.get("/api/data", name="API조회")
```

> **@task의 가중치란?**
> `@task(3)`과 `@task(1)`이 있으면, 메인페이지 조회가 API 조회보다 **3배 더 자주** 실행됩니다.
> 실제 사용 패턴에서 "조회가 많고, 쓰기가 적은" 비율을 반영할 때 유용합니다.

#### Step 2 — Locust 실행

```bash
# Docker를 이용한 실행
docker compose up locust
```

#### Step 3 — Web UI에서 테스트 시작

1. 브라우저에서 `http://localhost:8089` 접속
2. 아래 항목을 입력합니다:

| 항목 | 설명 | 예시 값 |
|------|------|--------|
| **Number of users** | 최대 가상 사용자 수 | 10 |
| **Ramp up** | 초당 추가되는 사용자 수 | 2 |
| **Host** | 테스트 대상 서버 주소 | http://target-server:8080 |

3. **Start** 버튼 클릭

#### Step 4 — 결과 확인

Web UI에서 실시간으로 확인할 수 있는 지표:

| 탭 | 확인 내용 |
|----|----------|
| **Statistics** | 각 API별 요청 수, 평균/중앙값/p95/p99 응답 시간, 에러율 |
| **Charts** | 응답 시간 추이, RPS 추이, 사용자 수 변화 그래프 |
| **Failures** | 실패한 요청의 상세 내용 (에러 메시지, 상태 코드) |

### 5.4 실습: 시나리오 기반 테스트

실제 사용자의 행동 흐름을 시뮬레이션합니다.

```python
from locust import HttpUser, task, between, SequentialTaskSet

class UserBehavior(SequentialTaskSet):
    """사용자 행동 시나리오: 로그인 → 목록 조회 → 상세 조회"""

    def on_start(self):
        """시나리오 시작 시 1회 실행: 로그인"""
        self.client.post("/api/login", json={
            "username": "testuser",
            "password": "testpass"
        }, name="1.로그인")

    @task
    def view_list(self):
        """상품 목록 조회"""
        self.client.get("/api/products", name="2.목록조회")

    @task
    def view_detail(self):
        """상품 상세 조회"""
        self.client.get("/api/products/1", name="3.상세조회")

class ScenarioUser(HttpUser):
    wait_time = between(2, 5)
    tasks = [UserBehavior]
```

> **SequentialTaskSet**을 사용하면 `@task`가 정의된 순서대로 실행됩니다.
> 일반 `@task`는 무작위 순서로 실행되지만, 로그인 → 조회 같은 흐름이 필요할 때 `SequentialTaskSet`을 사용합니다.

### 5.5 Locust 결과 읽는 법

테스트 완료 후 Statistics 탭의 주요 항목:

```
┌────────────┬─────────┬─────────┬─────────┬─────────┬──────────┐
│   Name     │ # Reqs  │ Avg(ms) │ p95(ms) │ p99(ms) │ Fail(%)  │
├────────────┼─────────┼─────────┼─────────┼─────────┼──────────┤
│ 메인페이지  │   450   │   120   │   250   │   800   │   0.2%   │
│ API조회    │   150   │   200   │   450   │  1200   │   0.5%   │
├────────────┼─────────┼─────────┼─────────┼─────────┼──────────┤
│ 합계       │   600   │   140   │   300   │   900   │   0.3%   │
└────────────┴─────────┴─────────┴─────────┴─────────┴──────────┘
```

**결과 해석 예시:**
- 전체 에러율 0.3%: 1% 이하이므로 **안정적**
- 메인페이지 p95 250ms: 95%의 사용자가 250ms 이내에 응답을 받음 → **양호**
- API조회 p99 1200ms: 1%의 사용자가 1.2초 이상 대기 → **개선 필요 여부 검토**

## 6. 성능 테스트 잘 해보기 (k6)

### 6.1 k6란?

k6는 **JavaScript(ES6) 기반의 현대적인 성능 테스트 도구**입니다.

| 항목 | 내용 |
|------|------|
| **언어** | JavaScript (ES6) |
| **특징** | CLI 기반, 코드로 모든 것을 정의, CI/CD 연동에 강점 |
| **장점** | 세밀한 시나리오 제어, 다양한 출력 형식, 임계값(Threshold) 자동 판정 |
| **공식 사이트** | k6.io |

### 6.2 Locust vs k6

| 비교 항목 | Locust | k6 |
|----------|--------|-----|
| **언어** | Python | JavaScript |
| **실행 방식** | Web UI + CLI | CLI 중심 |
| **UI** | Web UI 내장 | 외부 시각화(Grafana 연동) |
| **시나리오 제어** | 코드 기반 | 코드 기반 + Stage 설정 |
| **자동 판정** | 수동 확인 | **Thresholds로 자동 Pass/Fail** |
| **CI/CD 연동** | 가능 | **매우 용이** |
| **적합한 상황** | 빠른 프로토타이핑, Web UI 활용 | 정밀한 시나리오, 자동화 파이프라인 |

### 6.3 k6의 핵심 구조

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

// 테스트 설정
export const options = {
  // 가상 사용자 10명이 30초간 테스트
  vus: 10,
  duration: '30s',
};

// 테스트 시나리오 (각 VU가 반복 실행)
export default function () {
  const res = http.get('http://target-server:8080/');

  // 응답 검증
  check(res, {
    '상태코드 200': (r) => r.status === 200,
    '응답시간 500ms 이하': (r) => r.timings.duration < 500,
  });

  // Think Time: 1초 대기
  sleep(1);
}
```

| 구성 요소 | 설명 |
|----------|------|
| `options` | 테스트 설정 (VU 수, 지속 시간, 임계값 등) |
| `export default function` | 각 VU가 반복 실행하는 시나리오 |
| `check()` | 응답 값을 검증 (Pass/Fail 기록) |
| `sleep()` | Think Time 설정 |

### 6.4 실습: Stages를 활용한 부하 테스트

k6의 강점인 **Stages**로 Ramp-up → Steady → Ramp-down을 정밀하게 제어합니다.

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  // 부하 단계 설정
  stages: [
    { duration: '1m', target: 10 },   // 1분간 0 → 10명으로 증가 (Ramp-up)
    { duration: '3m', target: 10 },   // 3분간 10명 유지 (Steady)
    { duration: '1m', target: 0 },    // 1분간 10 → 0명으로 감소 (Ramp-down)
  ],

  // 성공 기준 (Thresholds)
  thresholds: {
    http_req_duration: [
      'p(95)<500',   // 95%의 요청이 500ms 이내
      'p(99)<1000',  // 99%의 요청이 1000ms 이내
    ],
    http_req_failed: [
      'rate<0.01',   // 에러율 1% 미만
    ],
  },
};

export default function () {
  // 메인 페이지 요청
  const res = http.get('http://target-server:8080/');

  // 응답 검증
  check(res, {
    '상태코드 200': (r) => r.status === 200,
  });

  // Think Time: 1~3초 랜덤 대기
  sleep(Math.random() * 2 + 1);
}
```

### 6.5 Thresholds — 자동 Pass/Fail 판정

k6의 가장 강력한 기능 중 하나는 **Thresholds(임계값)** 입니다.

```javascript
export const options = {
  thresholds: {
    // 응답 시간 기준
    http_req_duration: ['p(95)<500'],    // p95 응답 시간 500ms 이하

    // 에러율 기준
    http_req_failed: ['rate<0.01'],      // 에러율 1% 미만

    // 특정 요청에 대한 기준
    'http_req_duration{name:로그인}': ['p(95)<300'],  // 로그인 API p95 300ms 이하
  },
};
```

**실행 결과 예시:**

```
✓ http_req_duration............: avg=120ms  p(95)=350ms  p(99)=800ms
  ✓ { p(95)<500 } ──────────── pass ✓
✓ http_req_failed...............: 0.25%
  ✓ { rate<0.01 } ─────────── pass ✓
```

- Thresholds가 **모두 통과**하면 k6는 종료 코드 `0` (성공)을 반환
- **하나라도 실패**하면 종료 코드 `99` (실패)를 반환
- CI/CD 파이프라인에서 이 종료 코드로 **자동 합격/불합격 판정**이 가능합니다.

### 6.6 실습: 시나리오 기반 테스트

```javascript
import http from 'k6/http';
import { check, group, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '30s', target: 5 },
    { duration: '2m', target: 5 },
    { duration: '30s', target: 0 },
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],
    http_req_failed: ['rate<0.01'],
  },
};

export default function () {
  // 시나리오를 그룹으로 구분하여 결과를 체계적으로 확인
  group('01_로그인', function () {
    const loginRes = http.post(
      'http://target-server:8080/api/login',
      JSON.stringify({
        username: 'testuser',
        password: 'testpass',
      }),
      { headers: { 'Content-Type': 'application/json' } }
    );

    check(loginRes, {
      '로그인 성공': (r) => r.status === 200,
    });

    sleep(1);
  });

  group('02_목록조회', function () {
    const listRes = http.get('http://target-server:8080/api/products');

    check(listRes, {
      '목록 조회 성공': (r) => r.status === 200,
      '목록 데이터 존재': (r) => JSON.parse(r.body).length > 0,
    });

    sleep(2);
  });

  group('03_상세조회', function () {
    const detailRes = http.get('http://target-server:8080/api/products/1');

    check(detailRes, {
      '상세 조회 성공': (r) => r.status === 200,
    });

    sleep(1);
  });
}
```

> **group()** 을 사용하면 시나리오의 각 단계별로 성능 지표를 분리해서 확인할 수 있습니다.
> 어떤 단계에서 병목이 발생하는지 한눈에 파악할 수 있습니다.
