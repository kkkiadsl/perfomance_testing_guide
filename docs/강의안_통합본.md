## 섹션 A: 왜 성능 테스트를 해야 하는가? (12분, 5장)

### 슬라이드 A-1: 성능 테스트란? — 정의와 비유

핵심 메시지: 성능 테스트는 "수백, 수천 명이 동시에 사용해도 서버가 버틸 수 있는가?"를 확인하는 테스트이다.

설명 내용:
기능 테스트와 성능 테스트의 비교:
- 기능 테스트 — 핵심 질문: 올바르게 동작하는가? / 관심사: 정확성 / 사용자 수: 1명
- 성능 테스트 — 핵심 질문: 대규모 사용자가 동시에 써도 동작하는가? / 관심사: 속도, 안정성, 확장성 / 사용자 수: 수십~수만 명

엘리베이터 비유:
- 기능 테스트: 버튼을 누르면 해당 층으로 이동하는가?
- 성능 테스트: 정원 10명인 엘리베이터에 15명이 타면 어떻게 되는가?
- 핵심 포인트: 기능이 "정상"이어도, 부하가 걸리면 전혀 다른 결과가 나올 수 있다.

디자인 지시: 상단에 기능 테스트와 성능 테스트를 나란히 비교하는 2열 레이아웃. 하단에 엘리베이터 일러스트 — 왼쪽에 정상 탑승(기능 테스트), 오른쪽에 과부하 탑승(성능 테스트).

### 슬라이드 A-2: 성능 장애는 실제로 일어난다 ① — 예매 사이트

핵심 메시지: 스파이크 테스트를 하지 않으면 이벤트 오픈 시 장애가 발생한다.

설명 내용:
제목: "새벽에 열었는데 서버가 터졌어요"

상황: 인기 콘서트 티켓 오픈 시간에 수십만 명이 동시 접속했다.

증상:
- 페이지 로딩 시간 30초 이상
- "504 Gateway Timeout" 에러 대량 발생
- 결제 도중 세션 만료, 중복 결제 발생

원인:
- 스파이크 테스트 미실시: 순간 트래픽 급증 시나리오를 사전에 검증하지 않았다.
- 대기열(Queue) 시스템 부재

교훈: 이벤트성 서비스는 반드시 스파이크 테스트를 수행해야 하며, 예상 최대 동시 접속자의 2~3배 수준으로 테스트해야 한다.

디자인 지시: 왼쪽에 급증하는 트래픽 그래프, 오른쪽에 에러 화면 이미지를 배치.

### 슬라이드 A-3: 성능 장애는 실제로 일어난다 ② — 쇼핑몰과 사내 시스템

핵심 메시지: 전체 흐름을 테스트하지 않거나, 피크 시간대를 고려하지 않으면 장애가 발생한다.

설명 내용:

사례 1: "블랙프라이데이에 결제가 안 돼요"
- 상황: 대형 할인 행사에 평소 대비 10배 트래픽 발생. 상품 조회는 되지만 결제 API에서 병목 발생.
- 증상: 상품 조회 200ms(정상) / 결제 API 15,000ms → Timeout / 에러율 45%
- 원인: 부하 테스트 시 결제 API를 제외. "결제는 외부 PG사 영역"이라고 판단.
- 교훈: 전체 사용자 흐름(End-to-End)을 테스트해야 한다. 가장 약한 고리를 놓치면 안 된다.

사례 2: "월요일 아침마다 느려요"
- 상황: 월요일 오전 9시, 전 직원 동시 접속. 일주일 치 알림과 데이터가 한꺼번에 로딩.
- 증상: 대시보드 로딩 평소 2초 → 월요일 15초 / DB 커넥션 풀 고갈
- 원인: 특정 시간대 집중 접속 패턴을 고려하지 않음.
- 교훈: "평균 사용자 수"가 아닌 "피크 시간대의 동시 사용자 수"를 기준으로 설계해야 한다.

디자인 지시: 2분할 레이아웃. 각 사례를 좌우에 배치하고 교훈을 강조.

### 슬라이드 A-4: 성능 테스트 5대 원칙

핵심 메시지: 사례에서 배운 성능 테스트 5대 원칙

설명 내용:
1. 실제 사용 패턴을 반영한 시나리오를 설계하라
2. 예상 트래픽의 2~3배 수준으로 테스트하라
3. 전체 사용자 흐름(End-to-End)을 테스트 범위에 포함하라
4. 피크 시간대, 이벤트 등 특수 상황을 반드시 고려하라
5. 결과를 수치로 기록하고, 기준(Baseline)을 만들어라

디자인 지시: 5가지 원칙을 번호가 매겨진 카드 형태로 배치. 강조색으로 핵심 키워드 하이라이트.

### 슬라이드 A-5: QA 엔지니어가 성능 테스트를 알아야 하는 4가지 이유

핵심 메시지: QA의 역할은 기능 품질을 넘어 성능 품질까지 확장되고 있다.

설명 내용:
1. 품질의 완성 — 기능이 정상이어도 느리면 사용자는 떠난다. Google 연구: 페이지 로딩이 3초를 넘기면 모바일 사용자의 53%가 이탈.
2. 장애 예방 — 출시 전에 서버의 한계를 파악해야 장애를 예방할 수 있다. 위 3가지 사례 모두 사전 성능 테스트로 예방 가능했다.
3. 근거 기반 판단 — "느린 것 같아요"가 아닌, "p95 응답 시간이 2초를 초과합니다"처럼 수치로 품질을 이야기할 수 있다.
4. 커뮤니케이션 — "응답 시간이 느려요"가 아닌 "동시 사용자 500명 이상에서 에러율이 5%를 초과합니다"라고 말할 수 있다.

디자인 지시: 4개의 아이콘 카드 형태. 강조색으로 핵심 키워드 하이라이트.

## 섹션 B: 성능 테스트 4가지 유형 (10분, 4장)

### 슬라이드 B-1: 4가지 유형 한눈에 보기

핵심 메시지: 목적에 따라 4가지 유형으로 나뉜다. 각각 다른 질문에 답한다.

설명 내용:
1. 부하 테스트(Load Test)
   - 핵심 질문: "평소 트래픽을 잘 처리하는가?"
   - 방법: 예상 동시 사용자를 일정하게 유지
   - 패턴: 수평선 형태
   - 예: 평일 기준 동시 사용자 1,000명으로 30분간 테스트

2. 스트레스 테스트(Stress Test)
   - 핵심 질문: "어디까지 버틸 수 있는가?"
   - 방법: 사용자를 점진적으로 계속 증가
   - 패턴: 우상향 계단 형태
   - 예: 100명부터 시작해 10,000명까지 단계적으로 증가

3. 스파이크 테스트(Spike Test)
   - 핵심 질문: "갑자기 몰리면 어떻게 되는가?"
   - 방법: 짧은 시간 안에 극적으로 사용자 수 증가
   - 패턴: 급격한 피크 모양
   - 예: 100명에서 순간적으로 5,000명으로 급증

4. 내구성 테스트(Soak Test)
   - 핵심 질문: "오래 돌려도 괜찮은가?"
   - 방법: 적당한 부하를 장시간 유지
   - 패턴: 장시간 수평선 형태
   - 예: 동시 사용자 500명으로 4~8시간 연속 테스트

디자인 지시: 4가지 유형을 2×2 그리드로 배치. 각 유형 아래에 부하 패턴 미니 그래프(시간 축 대비 사용자 수 변화) 표현.

### 슬라이드 B-2: 부하 테스트 vs 스트레스 테스트 — 언제 쓰는가?

핵심 메시지: 부하 테스트는 "통과 기준 확인", 스트레스 테스트는 "한계 탐색"이다.

설명 내용:
부하 테스트 (Load Test):
- 목적: 이미 알고 있는 예상 트래픽에서 서버가 정상 동작하는지 검증
- 언제: 신규 서비스 오픈 전, 릴리스 전 회귀 검증
- 성공 기준: p95 응답시간 < 500ms, 에러율 < 1%
- 실습 파일: `01_load_test.py`

스트레스 테스트 (Stress Test):
- 목적: 서버가 버티지 못하는 지점을 찾는다. "지금 서버는 몇 명까지 처리할 수 있나?"
- 언제: 인프라 스펙을 결정하기 전, 용량 계획(Capacity Planning)
- 성공 기준: 한계점 수치 파악 자체가 목적 (테스트가 실패해도 성공)
- 실습 파일: `02_stress_test.py`

핵심 차이: 부하 테스트는 "이 선 안에서 괜찮은가?"를 확인. 스트레스 테스트는 "선이 어디까지인가?"를 탐색.

디자인 지시: 두 유형의 부하 패턴 그래프를 나란히 배치. 부하 테스트는 "기준선" 표시, 스트레스 테스트는 "한계점(Breaking Point)" 표시.

### 슬라이드 B-3: 스파이크 테스트 — 이벤트 오픈의 악몽

핵심 메시지: 스파이크 테스트는 "예측 불가능한 트래픽 폭발"에 대한 준비다.

설명 내용:
스파이크 테스트가 필요한 실제 상황:
- 티켓 오픈 (0시 정각에 수십만 명이 동시 접속)
- 블랙프라이데이 할인 시작 순간
- 뉴스/SNS에서 서비스가 갑자기 언급될 때

스파이크 테스트의 3가지 질문:
1. 스파이크 발생 시 서버가 살아 있는가? (에러율이 100%가 되지는 않는가?)
2. 스파이크가 끝난 후 서버가 정상으로 회복하는가?
3. 회복에 얼마나 걸리는가?

스파이크와 스트레스의 차이:
- 스트레스: 천천히 계단식으로 올라간다 (서버가 어느 정도 적응할 시간이 있다)
- 스파이크: 갑자기 10배, 100배로 폭발한다 (서버가 준비할 틈이 없다)

실습 파일: `03_spike_test.py`

디자인 지시: 스파이크 패턴 그래프 + 실제 티켓 오픈 사례 아이콘. 스파이크 구간에서 응답시간/에러율이 함께 치솟는 그래프 표현.

### 슬라이드 B-4: 내구성 테스트 — 조용히 찾아오는 장애

핵심 메시지: 내구성 테스트는 "시간이 지나면 나타나는 숨겨진 결함"을 찾는다.

설명 내용:
내구성 테스트가 잡아내는 결함들:
- 메모리 누수(Memory Leak): 처음에는 응답 시간 100ms → 4시간 후 5000ms. 메모리가 계속 쌓여 서버가 결국 OOM(Out of Memory)으로 죽는다.
- DB 커넥션 고갈: 커넥션이 제대로 반환되지 않아 시간이 지나면 새로운 요청을 처리하지 못한다.
- 파일 핸들 누수: 열린 파일/소켓이 닫히지 않아 운영체제 자원이 고갈된다.

내구성 테스트의 패턴:
- 초반(1시간): 응답시간 100ms, 에러율 0% → 정상
- 중반(3시간): 응답시간 200ms → 슬금슬금 증가 신호
- 후반(6시간): 응답시간 3000ms, 에러율 20% → 심각

실습 파일: `04_soak_test.py` (강의에서는 10분 단축 버전으로 진행)

디자인 지시: 시간에 따라 응답시간이 점점 올라가는 꺾은선 그래프. 메모리 사용량도 함께 증가하는 그래프를 옆에 배치.

## 섹션 C: 무엇을 측정하는가 — 핵심 지표 (12분, 5장)

### 슬라이드 C-1: 응답 시간과 백분위(Percentile)

핵심 메시지: 평균이 아닌 백분위(p95, p99)로 봐야 진짜 사용자 경험을 알 수 있다.

설명 내용:
응답 시간(Response Time):
- 클라이언트가 요청을 보내면, 서버가 처리한 후 응답을 돌려주는 전체 시간. 밀리초(ms) 단위로 측정.

주요 측정값:
- 평균(Average): 전체 응답 시간의 합 ÷ 요청 수
- 중앙값(Median, p50): 전체 요청 중 50%가 이 시간 이내에 완료
- p95: 95%의 요청이 이 시간 이내에 완료. 100명 중 95명의 경험을 대표.
- p99: 99%의 요청이 이 시간 이내에 완료. "극단적 상황의 사용자 경험"

왜 평균만으로는 부족한가?
- 요청 1~9번: 각각 100ms
- 요청 10번: 10,000ms (10초)
- 평균: 1,090ms (나쁘지 않아 보임)
- p99: 10,000ms (실제 1명은 10초를 기다렸다!)
- 평균 뒤에 숨은 10번째 사용자는 10초나 기다렸다. p99는 "100명 중 가장 불행한 1명의 경험"을 보여준다.

업계 기준:
- p95 500ms 이하 → 양호
- p95 1,000ms 이하 → 허용 가능
- p95 3,000ms 이상 → 불량

Amazon 연구: 응답 시간이 100ms 증가할 때마다 매출이 1% 감소.

디자인 지시: 상단에 클라이언트 → 서버 → 클라이언트 흐름 다이어그램. 하단에 10개 요청 막대 그래프(9개 낮은 막대 + 1개 극단적 막대). 평균선과 p99선을 표시하여 차이 강조.

### 슬라이드 C-2: 처리량 — RPS와 TPS

핵심 메시지: RPS는 "서버가 얼마나 빠르게 처리하는가"를 나타내는 핵심 지표이다.

설명 내용:
RPS (Requests Per Second):
- 초당 처리 요청 수
- 계산식: RPS = 총 요청 수 ÷ 총 소요 시간(초)
- 개별 HTTP 요청 1건 기준
- Locust UI의 "RPS" 칸에서 확인 가능

TPS (Transactions Per Second):
- 초당 처리 트랜잭션 수
- 비즈니스 트랜잭션 1건 기준
- 예: 로그인 → 상품 조회 → 결제 = 3개 HTTP 요청 = 1 Transaction
- RPS는 개별 요청, TPS는 업무 단위

실무에서 RPS와 TPS를 구분하는 이유:
- "결제 TPS가 100이다" = 초당 100건의 결제가 완료된다는 의미
- "결제 RPS가 500이다" = 결제 1건에 HTTP 요청이 5번 발생한다면 TPS는 100
- 개발팀/인프라팀과 커뮤니케이션 시 용어를 맞춰야 한다

디자인 지시: RPS와 TPS를 나란히 비교. 사용자가 "결제하기"를 클릭했을 때 내부에서 5개의 HTTP 요청이 발생하는 그림으로 차이를 설명.

### 슬라이드 C-3: 가상 사용자(VU)와 Think Time

핵심 메시지: 실제 사용자처럼 행동하는 가상 사용자를 만들어야 의미 있는 결과를 얻을 수 있다.

설명 내용:
가상 사용자 (Virtual User, VU):
- 성능 테스트 도구가 생성하는 가상의 사용자
- 각 VU는 독립적으로 시나리오를 반복 수행한다
- VU 10명 = 10명의 사용자가 동시에 같은 행동을 반복하는 것

Think Time (대기 시간):
- 실제 사용자는 페이지를 읽거나 입력하는 데 시간을 소비한다
- 예: 목록을 보고 3초 → 상품을 클릭하고 5초 → 결제하기 버튼 클릭
- Think Time 없이 테스트 → 실제보다 훨씬 많은 요청이 발생 → 비현실적 결과
- Locust에서는 `wait_time = between(1, 3)`으로 설정

Ramp-up:
- 가상 사용자를 한꺼번에 투입하지 않고 점진적으로 늘리는 과정
- 갑자기 100명을 투입하면 서버가 스파이크 상황이 되어 결과가 왜곡됨
- Locust Web UI의 "Spawn rate" = Ramp-up 속도 (초당 추가 사용자 수)

디자인 지시: 상단에 VU들이 병렬로 시나리오를 반복하는 모습. 하단에 Think Time이 포함된 사용자 행동 타임라인(요청 → 대기 → 요청 → 대기).

### 슬라이드 C-4: 에러율과 한계점

핵심 메시지: 에러율이 급증하는 지점이 서버의 한계점이다.

설명 내용:
에러율(Error Rate):
- 계산식: 에러율(%) = (실패 요청 수 ÷ 전체 요청 수) × 100
- 일반적으로 에러율 1% 이하를 안정적이라고 판단
- 에러율 5% 이상이면 사용자가 체감할 수 있는 수준

한계점(Breaking Point):
- 부하가 증가하면서 에러율이 급증하고 응답시간이 폭증하는 지점
- 스트레스 테스트의 목표가 바로 이 지점을 찾는 것

에러 유형 구분:
- 4xx (클라이언트 에러): 잘못된 요청, 권한 없음. 서버 문제가 아닌 경우가 많다.
- 5xx (서버 에러): 서버 내부 오류. 성능 테스트에서 추적해야 할 핵심 에러.
- Timeout: 지정한 시간 내에 응답이 오지 않은 경우. 서버 과부하 신호.

디자인 지시: 사용자 수 증가에 따라 에러율이 급증하는 그래프. 한계점에 빨간색 수직선 표시. "안전 구간" / "주의 구간" / "위험 구간" 색상으로 구분.

### 슬라이드 C-5: 테스트 구조 — Ramp-up / Steady / Ramp-down

핵심 메시지: 성능 테스트는 3단계로 구성된다. 점진적으로 부하를 올리고, 유지하고, 내린다.

설명 내용:
1. Ramp-up (점진적 증가)
   - 사용자를 점진적으로 증가시키는 구간
   - 갑작스러운 부하로 인한 결과 왜곡 방지
   - Locust에서: Spawn rate로 제어

2. Steady State (최대 부하 유지)
   - 목표 사용자 수를 유지하는 구간
   - 안정적인 성능 데이터를 수집하는 핵심 구간
   - 이 구간의 데이터가 테스트 결과의 핵심

3. Ramp-down (점진적 감소)
   - 사용자를 점진적으로 줄이는 구간
   - 부하 해소 후 시스템이 정상으로 복구되는지 확인

디자인 지시: 시간(X축) 대비 사용자 수(Y축)의 사다리꼴 그래프. Ramp-up=초록, Steady=파랑, Ramp-down=주황으로 색상 구분.

## 섹션 D: 실무에서의 성능 테스트 (10분, 4장)

### 슬라이드 D-1: 성능 테스트를 수행하는 시점

핵심 메시지: 성능 테스트는 "한 번 하고 끝"이 아니라, 개발 주기 안에서 반복적으로 수행한다.

설명 내용:
5가지 시점:
1. 신규 출시 전 — 서비스가 처음 세상에 나가기 전. 이때 기준선(Baseline)을 잡는다.
2. 주요 릴리스 전 — 새로운 기능 추가나 아키텍처 변경 시. 회귀 테스트 성격.
3. 이벤트·프로모션 전 — 트래픽 급증이 예상되는 이벤트 전. 스파이크 테스트 필수.
4. 인프라 변경 후 — 서버 스펙 변경, DB 마이그레이션 등 인프라 변경 후 검증.
5. 정기 회귀 테스트 — 주기적으로 자동 실행하여 성능 저하를 조기 발견.

디자인 지시: 개발 주기(기획→개발→QA→릴리스→운영) 타임라인에 각 시점의 성능 테스트 아이콘 배치.

### 슬라이드 D-2: 성능 테스트 환경

핵심 메시지: 환경에 따라 성능 테스트의 목적과 결과 해석이 달라진다.

설명 내용:
로컬 환경 (오늘 실습 환경):
- 목적: 도구 학습, 스크립트 검증
- 주의: 절대 수치를 신뢰하면 안 된다. 도구 사용법과 시나리오 검증이 목적.

스테이징 환경:
- 목적: 운영 배포 전 실제 성능 검증
- 특징: 운영 환경과 동일하거나 유사한 서버 스펙/DB/네트워크
- 실무에서 성능 테스트의 핵심 무대

운영 환경:
- 목적: 실제 인프라에서의 성능 확인
- 주의: 트래픽이 적은 시간대에 실행, 반드시 팀 합의 하에 진행

디자인 지시: 3개 환경을 좌→우로 배치. 신뢰도 화살표(로컬→스테이징→운영으로 갈수록 신뢰도 증가).

### 슬라이드 D-3: QA 엔지니어의 역할과 협업

핵심 메시지: 성능 테스트는 QA 혼자 하는 것이 아니다. QA가 주도하되 팀과 협업한다.

설명 내용:
QA 엔지니어의 역할:
- 테스트 시나리오 설계: 실제 사용자 패턴 분석
- 테스트 실행 및 모니터링: 도구 활용, 실시간 관찰
- 결과 분석 및 보고: 이슈 발견 후 팀에 공유
- 성능 기준 관리: Threshold, SLO 정의 및 지속 검증

협업 구조:
- 개발팀: QA가 "어디서 느린지" 데이터 제공 → 개발팀이 "왜 느린지" 분석
- 인프라팀: QA가 "한계점" 알려줌 → 인프라팀이 용량 조정
- 기획/PM: QA가 테스트 목표 설정을 위한 예상 트래픽 정보 수신

디자인 지시: 중앙에 QA 엔지니어, 주변에 3개 팀을 배치한 협업 다이어그램.

### 슬라이드 D-4: 성능 테스트 5단계 프로세스와 흔한 실수

핵심 메시지: 5단계 프로세스를 따르고, 흔한 실수를 피해야 한다.

설명 내용:
5단계 프로세스:
1. 목표 설정 — 대상 API, 목표 RPS, 성공 기준(Threshold) 정의
2. 시나리오 설계 — 실제 유저 흐름, 테스트 데이터, Think Time 설계
3. 환경 구성 — 대상 서버, 부하 생성기, 모니터링 도구 준비
4. 실행 및 모니터링 — 테스트 실행, 실시간 응답시간/RPS/에러율 관찰
5. 분석 및 보고 — 결과 해석, 병목 식별, 개발팀과 원인 분석

흔한 5가지 실수:
1. Think Time 없이 테스트 → 비현실적 결과
2. 단일 API만 테스트 → 실제 패턴과 다름
3. 로컬 환경 결과를 절대 기준으로 삼음 → 잘못된 판단
4. Ramp-up 없이 바로 최대 부하 → 왜곡된 결과
5. 1회만 실행 후 결론 → 최소 3회 반복 후 평균으로 판단

디자인 지시: 상단에 5단계 플로우차트, 하단에 "흔한 실수 TOP 5" 경고 리스트.

### 슬라이드 F-2: Locust 스크립트 구조 이해하기

핵심 메시지: Locust 스크립트는 4가지 핵심 요소로 구성된다.

설명 내용:

```python
from locust import HttpUser, task, between

class 쇼핑몰사용자(HttpUser):
    wait_time = between(1, 3)      # ① Think Time: 1~3초 대기

    @task(3)                       # ② 가중치 3 (상세보다 3배 빈번)
    def 상품_목록_조회(self):
        self.client.get("/api/items", name="상품 목록")  # ③

    @task(1)                       # ② 가중치 1
    def 상품_상세_조회(self):
        self.client.get("/api/items/1", name="상품 상세")
```

4가지 핵심 요소:
- ① `HttpUser`: 가상 사용자 클래스. 이것을 상속받아 시나리오 작성.
- ② `wait_time`: Think Time. between(1,3)은 1~3초 사이 랜덤 대기.
- ③ `@task(weight)`: 반복 실행할 작업. 숫자가 클수록 더 자주 실행.
- ④ `name` 파라미터: 동적 URL(`/items/1`, `/items/2`...)을 하나로 묶어서 집계.

발표자 노트: 수강생들이 코드를 직접 타이핑하도록 유도한다.

디자인 지시: 코드 블록과 함께 각 요소를 화살표로 가리키며 설명하는 주석 레이아웃.

### 슬라이드 F-3: Locust Web UI 구성

핵심 메시지: 3가지 탭에서 실시간으로 성능 지표를 확인한다.

설명 내용:
Statistics 탭:
- 각 API별 요청 수, RPS, 응답시간(평균/p50/p95/p99), 에러율 표시
- 확인 포인트: @task 가중치대로 요청 비율이 나오는가?

Charts 탭:
- 응답시간 추이 그래프: 시간이 지남에 따라 응답시간이 안정적인가? 증가하는가?
- RPS 추이 그래프: 초당 처리 요청 수

Failures 탭:
- 실패한 요청의 에러 메시지와 HTTP 상태 코드
- 어떤 API에서 에러가 발생하는지 파악

디자인 지시: 3개 탭의 화면 스크린샷을 탭 형태로 나란히 배치. 각 탭의 핵심 확인 포인트를 강조.


### 슬라이드 G-8: Locust 결과 읽기 — Statistics 탭 해석

핵심 메시지: Statistics 탭의 각 수치가 무엇을 의미하는지 함께 해석한다.

설명 내용:
Statistics 탭 결과 예시:

| Name | Reqs | Fails | Median | 95%ile | Avg | RPS |
|---|---|---|---|---|---|---|
| GET /api/items | 600 | 2 | 110ms | 280ms | 130ms | 3.2 |
| GET /api/items/[id] | 360 | 1 | 80ms | 200ms | 95ms | 2.1 |
| POST /api/items | 120 | 0 | 350ms | 680ms | 380ms | 0.7 |
| Aggregated | 1080 | 3 | 100ms | 280ms | 145ms | 6.0 |

함께 해석하기:
- 요청 비율 600:360:120 = 5:3:1 → @task 가중치(5:3:1)가 정확히 반영됨
- POST /api/items가 다른 API보다 3~4배 느림 → 쓰기 작업의 DB 부하 반영
- 전체 에러율 3/1080 = 0.28% → 1% 이하, 안정적
- p95 280ms → 95%의 사용자가 280ms 이내에 응답 받음 → 양호

"이 서버는 합격인가, 불합격인가?" 질문 → 성공 기준을 먼저 정해야 판단할 수 있다!

발표자 노트: 수치 해석 후 "이 서버는 합격인가요?" 질문 던지기. 정답은 "기준이 없으면 판단할 수 없다"는 것 강조.

디자인 지시: 테이블로 결과 표시, 핵심 수치에 색상 강조. "양호/검토필요" 뱃지.

- "3단계 상품 생성"이 "2단계 상세 조회"보다 3~4배 느린 이유는? (쓰기 vs 읽기 부하)

핵심 포인트: API를 분리해서 테스트하면 "어느 API가 병목인가?"를 정확히 찾을 수 있다.
실제 업무에서도: "전체 API가 느린 것인가?" vs "특정 API 하나가 문제인가?"를 먼저 구분해야 한다.

디자인 지시: API별 응답시간 비교 표 + "병목 API" 강조 표시.

### 슬라이드 H-3: 결과를 어떻게 보고할 것인가?

핵심 메시지: 성능 테스트 결과를 팀에 공유할 때 이 형식을 사용한다.

설명 내용:
보고 형식 예시:

**성능 테스트 결과 요약**
- 테스트 일시: 2026-03-24
- 대상: 상품 API (목록/상세/생성)
- 환경: 로컬 (Locust + FastAPI)

**결과**
| 지표 | 결과 | 기준 | 판정 |
|---|---|---|---|
| p95 응답시간 | 280ms | < 500ms | ✅ 통과 |
| 에러율 | 0.28% | < 1% | ✅ 통과 |
| 최대 안정 사용자 | 60명 | - | 참고 |

**이슈**
- POST /api/items가 GET API 대비 3~4배 느림. 쓰기 부하 최적화 검토 필요.

발표자 노트: "수치로 말하는 QA"의 실제 예시를 보여준다. "응답이 느린 것 같아요" 대신 "p95가 기준치를 초과합니다"라고 말하는 연습.

디자인 지시: 보고 형식 템플릿을 카드 형태로 표현. "✅ 통과" / "❌ 실패" 뱃지 강조.

## 섹션 I: 핵심 용어 암기 카드 (5분, 2장)

### 슬라이드 I-1: 반드시 알아야 할 용어 10선 (앞면)

핵심 메시지: 이 10개 단어를 알면 성능 테스트 커뮤니케이션의 90%가 해결된다.

설명 내용:
10가지 핵심 용어:

1. VU (Virtual User) — 가상 사용자. 도구가 생성하는 시뮬레이션 사용자 수.
2. Think Time — 요청 사이 대기 시간. 실제 사용자의 행동 패턴 반영.
3. Ramp-up — 가상 사용자를 점진적으로 증가시키는 단계.
4. RPS (Requests Per Second) — 초당 처리 요청 수. 서버의 처리 능력.
5. TPS (Transactions Per Second) — 초당 처리 트랜잭션 수. 비즈니스 처리 능력.
6. p95 / p99 — 95%, 99% 백분위 응답시간. 대부분의 사용자 경험을 대표.
7. 에러율 (Error Rate) — 전체 요청 중 실패 비율. 1% 이하가 안정적.
8. Baseline — 기준 성능 수치. 이후 비교의 기준점.
9. Breaking Point — 한계점. 에러율이 급증하고 응답시간이 폭증하는 지점.
10. Threshold — 성공 기준. "p95 < 500ms"처럼 합격/불합격 기준을 정의한 수치.

발표자 노트: 슬라이드를 보지 않고 10개를 말할 수 있다면 성능 테스트 입문자 자격증 획득!

디자인 지시: 10개 용어를 2열 5행 카드 그리드로 배치. 각 카드에 용어(강조)와 한 줄 설명. 번호로 순서 표시.

### 슬라이드 I-2: 용어 활용 — 실무 커뮤니케이션 문장

핵심 메시지: 용어를 알면 이렇게 말할 수 있다.

설명 내용:
전 vs 후 비교:

Before (용어를 모를 때):
- "서버가 느려진 것 같아요."
- "사람이 많이 몰리면 터질 것 같은데요."
- "응답이 빠른 것 같은데 가끔 느릴 때도 있어요."

After (용어를 알 때):
- "동시 사용자 60명 이상에서 p95 응답시간이 1,000ms를 초과합니다."
- "스파이크 테스트에서 순간 트래픽 급증 시 에러율이 42%까지 치솟습니다."
- "평균은 130ms이지만 p99는 3,200ms입니다. 일부 사용자가 3초 이상 대기하고 있습니다."

실무에서 TPS를 쓰는 예시:
- "현재 결제 TPS가 50인데, 블랙프라이데이에 예상 TPS는 300입니다. 지금 인프라로는 부족합니다."

발표자 노트: Before/After를 읽어주며 "이 차이가 QA 엔지니어로서의 전문성을 보여줍니다"라고 강조.

디자인 지시: Before(회색/흐릿한 말풍선)와 After(색상 있는 말풍선)를 나란히 배치. 핵심 용어를 굵게 강조.

# 부록 A: k6 심화 — JavaScript 기반 자동화 성능 테스트

> 이 섹션은 강의 본문에서 다루지 않는다.
> Locust 이후 더 심화된 자동화가 필요한 분들을 위한 참고 자료이다.

## k6 소개

k6는 JavaScript 기반의 성능 테스트 도구로, CI/CD 파이프라인 연동과 자동 Pass/Fail 판정에 강점이 있다.

- 언어: JavaScript (ES6)
- 특징: CLI 기반, Thresholds로 자동 합격/불합격 판정
- 장점: CI/CD 연동 용이 (종료 코드로 자동 판정), 세밀한 Stages 제어
- 공식 사이트: k6.io

## Locust vs k6 비교

| 항목 | Locust | k6 |
|---|---|---|
| 언어 | Python | JavaScript |
| 실행 방식 | Web UI + CLI | CLI 중심 |
| 시각화 | Web UI 내장 | Grafana 연동 |
| 자동 판정 | 수동 확인 | Thresholds로 자동 Pass/Fail |
| CI/CD 연동 | 가능 | 종료 코드로 매우 용이 |
| 적합한 상황 | 빠른 프로토타이핑, 탐색 | 자동화, 반복 회귀 테스트 |

## k6 기본 스크립트 구조

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '30s', target: 5 },   // Ramp-up
    { duration: '1m',  target: 5 },   // Steady
    { duration: '30s', target: 0 },   // Ramp-down
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],  // p95 응답시간 500ms 이하
    http_req_failed:   ['rate<0.01'],  // 에러율 1% 미만
  },
};

export default function () {
  const res = http.get('http://localhost:8080/api/items');
  check(res, { '상태코드 200': (r) => r.status === 200 });
  sleep(Math.random() * 2 + 1);  // 1~3초 Think Time
}
```

# 부록 C: 성능 테스트 관련 참고 근거 자료

이 섹션은 강의 자료의 근거가 되는 연구 및 업계 기준이다.

## 응답 시간과 사용자 경험

- Google/SOASTA 연구 (2017): 모바일 페이지 로딩 1→3초 시 이탈률 32% 증가
- Amazon 내부 연구 (Greg Linden, 2006): 응답시간 100ms 증가 시 매출 1% 감소
- Akamai 연구: 로딩 2초 초과 시 사용자의 47% 이탈

## 업계 표준 성능 기준

- 응답시간: p95 기준 500ms 이하 양호, 1초 이하 허용, 3초 이상 불량
- 에러율: 1% 이하 안정적, 0.1% 이하 이상적
- 가용성: 99.9% = 연간 8.76시간 다운타임 허용

## 주요 성능 테스트 도구 생태계

- Apache JMeter: Java 기반, 가장 역사 오래됨, GUI 지원
- Gatling: Scala 기반, 상세 HTML 리포트
- Locust: Python 기반, Web UI 내장, 분산 지원
- k6: JavaScript 기반, CLI/CI·CD 최적화 (Grafana Labs 관리)
- Artillery: Node.js 기반, YAML 시나리오
- nGrinder: Naver 개발, 국내 기업 다수 사용
