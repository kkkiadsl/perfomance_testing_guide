## 섹션 A: 왜 성능 테스트를 해야 하는가? (12분, 5장)

### 슬라이드 A-1: 성능 테스트란? — 정의와 비유

핵심 메시지: 성능 테스트는 "수백, 수천 명이 동시에 사용해도 서버가 버틸 수 있는가?"를 확인하는 테스트이다.

설명 내용:
기능 테스트는 "이 기능이 올바르게 동작하는가?"를 확인한다. 성능 테스트는 여기서 한 걸음 더 나아가 "수백, 수천 명이 동시에 사용해도 올바르게 동작하는가?"를 확인한다.

기능 테스트와 성능 테스트의 비교:
- 기능 테스트 — 핵심 질문: 올바르게 동작하는가? / 관심사: 정확성 / 사용자 수: 1명
- 성능 테스트 — 핵심 질문: 대규모 사용자가 동시에 써도 동작하는가? / 관심사: 속도, 안정성, 확장성 / 사용자 수: 수십~수만 명

엘리베이터 비유:
- 기능 테스트: 버튼을 누르면 해당 층으로 이동하는가?
- 성능 테스트: 정원 10명인 엘리베이터에 15명이 타면 어떻게 되는가?
- 핵심 포인트: 기능이 "정상"이어도, 부하가 걸리면 전혀 다른 결과가 나올 수 있다.

디자인 지시: 상단에 기능 테스트와 성능 테스트를 나란히 비교하는 2열 레이아웃. 하단에 엘리베이터 일러스트 — 왼쪽에 정상 탑승(기능 테스트), 오른쪽에 과부하 탑승(성능 테스트).

### 슬라이드 A-2: 성능 장애는 실제로 일어난다 ① — 예매 사이트

핵심 메시지: 스파이크 테스트를 하지 않으면 이벤트 오픈 시 장애가 발생한다.

설명 내용:
제목: "새벽에 열었는데 서버가 터졌어요"

상황: 인기 콘서트 티켓 오픈 시간에 수십만 명이 동시 접속했다. 서버가 감당할 수 없는 트래픽이 순간적으로 몰렸다.

증상:
- 페이지 로딩 시간 30초 이상
- "504 Gateway Timeout" 에러 대량 발생
- 결제 도중 세션 만료, 중복 결제 발생

원인:
- 스파이크 테스트 미실시: 순간 트래픽 급증 시나리오를 사전에 검증하지 않았다.
- 대기열(Queue) 시스템 부재: 동시 접속을 제어하는 장치가 없었다.

교훈: 이벤트성 서비스는 반드시 스파이크 테스트를 수행해야 하며, 예상 최대 동시 접속자의 2~3배 수준으로 테스트해야 한다.

디자인 지시: 왼쪽에 급증하는 트래픽 그래프, 오른쪽에 에러 화면 이미지를 배치.

### 슬라이드 A-3: 성능 장애는 실제로 일어난다 ② — 쇼핑몰과 사내 시스템

핵심 메시지: 전체 흐름을 테스트하지 않거나, 피크 시간대를 고려하지 않으면 장애가 발생한다.

설명 내용:

사례 1: "블랙프라이데이에 결제가 안 돼요"
- 상황: 대형 할인 행사에 평소 대비 10배 트래픽 발생. 상품 조회는 되지만 결제 API에서 병목 발생.
- 증상: 상품 조회 200ms(정상) / 결제 API 15,000ms → Timeout / 에러율 45%
- 원인: 부하 테스트 시 결제 API를 제외. "결제는 외부 PG사 영역"이라고 판단.
- 교훈: 전체 사용자 흐름(End-to-End)을 테스트해야 한다. 가장 약한 고리를 놓치면 안 된다.

사례 2: "월요일 아침마다 느려요"
- 상황: 월요일 오전 9시, 전 직원 동시 접속. 일주일 치 알림과 데이터가 한꺼번에 로딩.
- 증상: 대시보드 로딩 평소 2초 → 월요일 15초 / DB 커넥션 풀 고갈 / 화요일 복구
- 원인: 특정 시간대 집중 접속 패턴을 고려하지 않음. DB 커넥션 풀 크기 부족.
- 교훈: "평균 사용자 수"가 아닌 "피크 시간대의 동시 사용자 수"를 기준으로 설계해야 한다.

디자인 지시: 2분할 레이아웃. 왼쪽에 쇼핑몰 사례(체인 형태의 사용자 흐름에서 결제 부분에 X 표시), 오른쪽에 사내 시스템 사례(요일별 트래픽 그래프에서 월요일 피크 강조).

### 슬라이드 A-4: 3가지 사례에서 배우는 성능 테스트 5대 원칙

핵심 메시지: 성능 테스트 5대 원칙

설명 내용:
1. 실제 사용 패턴을 반영한 시나리오를 설계하라
2. 예상 트래픽의 2~3배 수준으로 테스트하라
3. 전체 사용자 흐름(End-to-End)을 테스트 범위에 포함하라
4. 피크 시간대, 이벤트 등 특수 상황을 반드시 고려하라
5. 결과를 수치로 기록하고, 기준(Baseline)을 만들어라

디자인 지시: 5가지 원칙을 번호가 매겨진 카드 형태로 배치. 강조색을 사용하여 핵심 키워드를 하이라이트.

### 슬라이드 A-5: QA 엔지니어가 성능 테스트를 알아야 하는 4가지 이유

핵심 메시지: QA의 역할은 기능 품질을 넘어 성능 품질까지 확장되고 있다.

설명 내용:
1. 품질의 완성 — 기능이 정상이어도 느리면 사용자는 떠난다. Google의 연구에 따르면 페이지 로딩이 3초를 넘기면 모바일 사용자의 53%가 이탈한다.
2. 장애 예방 — 출시 전에 서버의 한계를 파악해야 장애를 예방할 수 있다. 위에서 본 3가지 사례 모두 사전 성능 테스트로 예방할 수 있었다.
3. 근거 기반 판단 — "느린 것 같아요"가 아닌, "p95 응답 시간이 2초를 초과합니다"처럼 수치로 품질을 이야기할 수 있다.
4. 커뮤니케이션 — 개발팀과 성능 이슈를 정량적으로 논의할 수 있다. "응답 시간이 느려요"가 아닌 "동시 사용자 500명 이상에서 에러율이 5%를 초과합니다"라고 말할 수 있다.

디자인 지시: 4개의 아이콘 카드 형태로 각 이유를 표현. 강조색으로 핵심 키워드 하이라이트.

## 섹션 B: 무엇을 테스트하고 측정하는가? (13분, 5장)

### 슬라이드 B-1: 성능 테스트의 4가지 종류

핵심 메시지: 목적에 따라 4가지 유형으로 나뉜다. 각각 다른 질문에 답한다.

설명 내용:
1. 부하 테스트(Load Test) — 예상 트래픽에서 정상 동작하는지 확인. 핵심 질문: "평소 트래픽을 잘 처리하는가?" 예: 평일 기준 동시 사용자 1,000명으로 30분간 테스트.
2. 스트레스 테스트(Stress Test) — 시스템의 한계점(Breaking Point)을 파악. 핵심 질문: "어디까지 버틸 수 있는가?" 예: 사용자를 100명부터 시작해 10,000명까지 점진적으로 증가.
3. 스파이크 테스트(Spike Test) — 갑작스러운 트래픽 급증에 대한 대응력 확인. 핵심 질문: "갑자기 몰리면 어떻게 되는가?" 예: 100명에서 순간적으로 5,000명으로 급증.
4. 내구성 테스트(Soak Test) — 장시간 운영 시 메모리 누수 등 안정성 확인. 핵심 질문: "오래 돌려도 괜찮은가?" 예: 동시 사용자 500명으로 4시간 연속 테스트.

디자인 지시: 부하 강도가 왼쪽에서 오른쪽으로 증가하는 4단계 다이어그램. 각 유형별로 부하 패턴 그래프(시간 축 대비 사용자 수 변화)를 작은 아이콘으로 표현.
- Load: 수평선 형태 (일정한 부하)
- Stress: 우상향 계단 (계속 증가하는 부하)
- Spike: 급격한 피크 모양 (순간 급증)
- Soak: 장시간 수평선 (오래 지속되는 부하)

### 슬라이드 B-2: 핵심 지표 ① — 응답 시간과 백분위(Percentile)

핵심 메시지: 평균이 아닌 백분위(p95, p99)로 봐야 진짜 사용자 경험을 알 수 있다.

설명 내용:
응답 시간(Response Time):
- 클라이언트가 요청(Request)을 보내면, 서버가 처리한 후 응답(Response)을 돌려준다. 이 전체 시간을 밀리초(ms) 단위로 측정한다.

주요 측정값:
- 평균(Average): 전체 응답 시간의 평균
- 중앙값(Median, p50): 전체 요청 중 50%가 이 시간 이내에 완료
- p90 / p95 / p99: 각각 90%, 95%, 99%의 요청이 이 시간 이내에 완료

왜 평균만으로는 부족한가? — 예시 데이터:
- 요청 1~9번: 각각 100ms
- 요청 10번: 10,000ms (10초)
- 결과: 평균은 1,090ms(나쁘지 않아 보임), p90은 100ms, p99는 10,000ms
- 평균 뒤에 숨은 10번째 사용자는 10초나 기다렸다. p99는 "100명 중 가장 불행한 1명의 경험"을 보여준다.

업계 근거: Amazon의 연구에서 응답 시간이 100ms 증가할 때마다 매출이 1% 감소한다는 결과가 나왔다. Google은 검색 결과 로딩이 0.5초 늘어나면 트래픽이 20% 감소한다고 발표했다.

디자인 지시: 상단에 클라이언트 → 서버 → 클라이언트 흐름 다이어그램. 하단에 10개 요청의 응답 시간을 막대 그래프로 시각화(9개 낮은 막대 + 1개 극단적 막대). 평균선과 p99선을 표시하여 차이를 강조.

### 슬라이드 B-3: 핵심 지표 ② — 처리량, 동시 사용자, 에러율

핵심 메시지: RPS는 서버의 처리 능력, 동시 사용자는 부하의 크기, 에러율은 한계의 신호이다.

설명 내용:
처리량 지표:
- RPS(Requests Per Second): 초당 처리 요청 수. 계산식: RPS = 총 요청 수 / 총 소요 시간(초)
- TPS(Transactions Per Second): 초당 처리 트랜잭션 수
- RPS는 개별 HTTP 요청 1건 기준이고, TPS는 비즈니스 트랜잭션 1건 기준이다. 예: 로그인 → 상품조회 → 결제라는 3개 HTTP 요청이 1 Transaction이 될 수 있다.

동시 사용자 수(Concurrent Users):
- 같은 시간대에 서버에 요청을 보내고 있는 사용자 수
- 전체 가입자 수나 DAU(일일 활성 사용자)와는 다른 개념. 실제로 "지금 이 순간" 요청을 보내고 있는 사용자만 카운트.

에러율(Error Rate):
- 계산식: 에러율(%) = (실패 요청 수 / 전체 요청 수) × 100
- 일반적으로 에러율 1% 이하를 안정적이라고 판단
- 에러율이 급증하는 지점이 시스템의 한계점

디자인 지시: 3개 지표(RPS, 동시 사용자, 에러율)를 각각의 게이지 또는 미터 형태로 시각화.

### 슬라이드 B-4: 핵심 개념 — 가상 사용자(VU)와 Think Time

핵심 메시지: 실제 사용자처럼 행동하는 가상 사용자를 만들어야 의미 있는 결과를 얻을 수 있다.

설명 내용:
가상 사용자 (Virtual User, VU):
- 성능 테스트 도구가 생성하는 가상의 사용자
- 각 VU는 독립적으로 시나리오를 반복 수행한다.
- VU 10명 = 10명의 사용자가 동시에 같은 행동을 반복하는 것

Think Time (대기 시간):
- 실제 사용자는 페이지를 읽거나 입력하는 데 시간을 소비한다.
- 예: 로그인 후 3초 대기 → 상품목록 조회 후 5초 대기 → 상품 상세 조회
- Think Time 없이 테스트하면 비현실적으로 높은 부하가 발생하여 잘못된 결과가 나온다.
- Think Time을 포함하면 실제 사용자 패턴에 가까운 부하를 생성할 수 있다.

디자인 지시: 상단에 여러 VU가 병렬로 시나리오를 반복하는 모습을 표현. 하단에 Think Time이 포함된 사용자 행동 타임라인 다이어그램(요청 → 대기 → 요청 → 대기 흐름).

### 슬라이드 B-5: 테스트 구조 — Ramp-up / Steady / Ramp-down

핵심 메시지: 성능 테스트는 3단계로 구성된다. 점진적으로 부하를 올리고, 유지하고, 내린다.

설명 내용:
1. Ramp-up (점진적 증가) — 사용자를 점진적으로 증가시키는 구간. 갑작스러운 부하로 인한 결과 왜곡을 방지하기 위한 목적이다.
2. Steady State (최대 부하 유지) — 목표 사용자 수를 유지하는 구간. 안정적인 성능 데이터를 수집하기 위한 핵심 구간이다.
3. Ramp-down (점진적 감소) — 사용자를 점진적으로 줄이는 구간. 부하 해소 후 시스템이 정상으로 복구되는지 확인한다.

디자인 지시: 시간(X축) 대비 사용자 수(Y축)의 사다리꼴 그래프. Ramp-up 구간은 초록색, Steady 구간은 파란색, Ramp-down 구간은 주황색으로 색상 구분.

## 섹션 C: 실무에서의 성능 테스트 — 시점, 환경, 역할 (10분, 4장)

### 슬라이드 C-1: 성능 테스트를 수행하는 시점

핵심 메시지: 성능 테스트는 "한 번 하고 끝"이 아니라, 개발 주기 안에서 반복적으로 수행한다.

설명 내용:
성능 테스트를 수행해야 하는 5가지 시점:

1. 신규 출시 전 — 서비스가 처음 세상에 나가기 전에 서버가 예상 트래픽을 감당할 수 있는지 확인한다. 이 시점에 기준선(Baseline)을 잡는다.
2. 주요 릴리스 전 — 새로운 기능이 추가되거나 아키텍처가 변경될 때마다 기존 성능에 영향이 없는지 확인한다. 회귀 테스트 성격이다.
3. 이벤트·프로모션 전 — 트래픽 급증이 예상되는 이벤트(세일, 티켓 오픈, 공채 등) 전에 스파이크 테스트와 부하 테스트를 수행한다.
4. 인프라 변경 후 — 서버 스펙 변경, DB 마이그레이션, 클라우드 이전 등 인프라가 바뀌면 성능 영향을 확인한다.
5. 정기 회귀 테스트 — 주기적으로(월 1회, 스프린트마다 등) 성능 테스트를 자동 실행하여 성능 저하를 조기에 발견한다. CI/CD 파이프라인에 통합하면 더욱 효과적이다.

디자인 지시: 개발 주기(기획 → 개발 → QA → 릴리스 → 운영)를 원형 또는 직선 타임라인으로 표현하고, 각 시점에 성능 테스트 아이콘을 배치.

### 슬라이드 C-2: 성능 테스트 환경

핵심 메시지: 환경에 따라 성능 테스트의 목적과 결과 해석이 달라진다.

설명 내용:
3가지 테스트 환경 비교:

로컬 환경 (개발자 PC / 이 강의의 실습 환경):
- 목적: 도구 학습, 스크립트 검증, 기본 동작 확인
- 특징: 자원이 제한적(CPU, 메모리, 네트워크 모두 한 대의 PC에서 동작)
- 주의: 절대 수치를 신뢰하면 안 된다. "도구 사용법 익히기"와 "시나리오 검증"이 목적이다.
- 이 강의의 실습이 바로 이 환경이다. Docker로 모든 구성 요소를 로컬에서 실행한다.

스테이징 환경 (운영과 유사한 별도 환경):
- 목적: 운영 배포 전 실제 성능 검증
- 특징: 운영 환경과 동일하거나 유사한 서버 스펙, DB, 네트워크 구성
- 주의: 운영과 데이터 규모, 외부 연동이 다를 수 있으므로 차이를 인지해야 한다.
- 실무에서 성능 테스트의 핵심 무대이다.

운영(프로덕션) 환경:
- 목적: 실제 인프라에서의 성능 확인
- 특징: 실제 사용자와 공유하는 환경이므로 서비스 영향을 최소화해야 한다.
- 주의: 반드시 트래픽이 적은 시간대(새벽 등)에 실행하고, 팀 합의 하에 진행한다.

디자인 지시: 3개 환경을 좌→우로 배치하고, 각 환경의 아이콘(노트북/서버 랙/클라우드)과 함께 목적·주의사항을 정리. 신뢰도 화살표(로컬→스테이징→운영으로 갈수록 신뢰도 증가)를 추가.

### 슬라이드 C-3: QA 엔지니어의 역할과 협업

핵심 메시지: 성능 테스트는 QA 혼자 하는 것이 아니다. QA가 주도하되, 개발팀·인프라팀과 협업한다.

설명 내용:
QA 엔지니어의 역할 — 성능 테스트의 기획자이자 실행자:
- 테스트 시나리오 설계: 실제 사용자 패턴을 분석하여 테스트 시나리오를 만든다.
- 테스트 실행 및 모니터링: 도구를 활용하여 테스트를 실행하고 실시간으로 관찰한다.
- 결과 분석 및 보고: 테스트 결과를 해석하고, 이슈를 발견하면 팀에 공유한다.
- 성능 기준 관리: Thresholds, SLO 등 성능 기준을 정의하고 지속적으로 검증한다.

협업 구조:
- 개발팀: 병목 원인 분석, 코드 레벨 최적화, API 설계 논의. QA가 "어디서 느려지는지" 데이터를 제공하면, 개발팀이 "왜 느린지"를 분석한다.
- 인프라/DevOps팀: 서버 스케일링, 리소스 모니터링, 환경 구성. QA가 "한계점"을 알려주면, 인프라팀이 용량을 조정한다.
- 기획/PM: 예상 트래픽 규모, 이벤트 일정, 성능 요구사항 정의. QA가 테스트 목표를 설정하기 위한 입력값을 제공받는다.

디자인 지시: 중앙에 QA 엔지니어, 주변에 개발팀/인프라팀/기획팀을 배치한 협업 다이어그램. 각 화살표에 주고받는 정보(시나리오, 병목 데이터, 인프라 스펙, 트래픽 예측)를 표시.

### 슬라이드 C-4: 성능 테스트 5단계 프로세스와 흔한 실수

핵심 메시지: 5단계 프로세스를 따르고, 흔한 실수를 피해야 한다.

설명 내용:
5단계 프로세스:
1단계. 목표 설정 — 대상 API, 목표 RPS, 성공 기준(Thresholds)을 정한다. 기획/PM에게 예상 트래픽 정보를 받는다.
2단계. 시나리오 설계 — 실제 유저 흐름, 테스트 데이터, Think Time을 설계한다. 로그 분석이나 GA 데이터를 참고한다.
3단계. 환경 구성 — 테스트 대상 서버, 부하 생성기, 모니터링 도구를 준비한다. 인프라팀과 협의한다.
4단계. 실행 및 모니터링 — 테스트를 실행하고 실시간으로 응답 시간, RPS, 에러율을 관찰한다.
5단계. 분석 및 보고 — 결과를 해석하고 병목 지점을 식별한다. 개발팀과 원인을 분석하고 개선 방향을 논의한다.

흔한 5가지 실수:
1. Think Time 없이 테스트 → 비현실적 결과. 올바른 접근: 실제 사용자 패턴을 반영한 Think Time 설정
2. 단일 API만 테스트 → 실제 패턴과 다름. 올바른 접근: 사용자 시나리오 기반으로 여러 API 조합
3. 로컬 환경 결과를 절대 기준으로 삼음 → 잘못된 판단. 올바른 접근: 운영 환경과 유사한 환경에서 실행
4. Ramp-up 없이 바로 최대 부하 → 왜곡된 결과. 올바른 접근: 점진적으로 부하 증가
5. 1회만 실행 후 결론 → 신뢰도 낮음. 올바른 접근: 최소 3회 반복 후 평균으로 판단

디자인 지시: 상단에 5단계 프로세스를 화살표 플로우차트로 표현. 하단에 "흔한 실수 TOP 5"를 경고 아이콘과 함께 리스트로 표현.

핵심 메시지: JavaScript 기반, 자동 Pass/Fail 판정과 CI/CD 연동에 강한 성능 테스트 도구

설명 내용:
k6 핵심 정보:
- 언어: JavaScript (ES6)
- 특징: CLI 기반, 코드로 모든 것을 정의
- 장점: 세밀한 시나리오 제어, 자동 Pass/Fail 판정(Thresholds), CI/CD 연동 용이
- 공식 사이트: k6.io
- 적합한 상황: 자동화 파이프라인, 정밀한 성능 기준 관리, 반복적인 회귀 테스트

디자인 지시: k6 로고와 함께 핵심 특징을 아이콘 리스트로 표현.

### 슬라이드 D-3: Locust vs k6 한눈에 비교

핵심 메시지: 두 도구 모두 훌륭하다. 팀 상황에 맞게 선택하면 된다.

설명 내용:
비교표:
- 언어: Locust=Python / k6=JavaScript
- 실행 방식: Locust=Web UI + CLI / k6=CLI 중심
- 시각화: Locust=Web UI 내장 / k6=Grafana 연동
- 시나리오 제어: Locust=코드 기반 / k6=코드 기반 + Stages
- 자동 판정: Locust=수동 확인 / k6=Thresholds로 자동 Pass/Fail
- CI/CD 연동: Locust=가능 / k6=매우 용이

오늘 실습 순서: 먼저 Locust로 성능 테스트의 기본 개념을 체험한 뒤, k6로 더 정밀한 테스트를 경험한다.

디자인 지시: 2열 비교 테이블. 각 항목에 아이콘을 붙여 시각적으로 구분.

## 섹션 E: 환경 확인 및 셋업 (5분, 2장)

### 슬라이드 E-1: 실습 아키텍처

핵심 메시지: 오늘 실습에서 사용할 전체 구성도

설명 내용:
실습 아키텍처의 구성요소와 데이터 흐름:

**대상 서버 (target-server/):**
- FastAPI 기반 샘플 API 서버이다. 별도의 docker-compose로 실행한다.
- http://localhost:8080 에 접속하면 실시간 모니터링 대시보드가 표시된다.
- 주요 API: 상품 목록(/api/items), 상품 상세(/api/items/{id}), 상품 생성(POST /api/items), 지연 시뮬레이션(/api/slow), 에러 시뮬레이션(/api/error)
- Locust와 k6는 Docker 컨테이너 내부에서 host.docker.internal:8080 으로 대상 서버에 접근한다.

**Locust 데이터 흐름 (locust/):**
- Locust → 대상 서버 (HTTP 요청으로 부하 생성)
- Locust는 자체 Web UI(http://localhost:8089)를 내장하고 있어 별도의 시각화 도구가 필요 없다.
- 테스트 실행과 동시에 웹 서버가 구동되며, 브라우저에서 실시간 모니터링(응답 시간, RPS, 에러율 등)이 가능하다.

**k6 데이터 흐름 (k6/):**
- k6 → 대상 서버 (HTTP 요청으로 부하 생성)
- k6 → InfluxDB (테스트 결과 지표를 시계열 데이터베이스에 저장)
- InfluxDB → Grafana (저장된 데이터를 그래프와 차트로 시각화)
- k6는 CLI 기반이므로 터미널에서 결과를 확인하거나, Grafana 대시보드(http://localhost:3001)를 통해 시각화된 결과를 분석한다.

각 구성요소(대상 서버, Locust, k6)는 별도의 docker-compose.yml로 관리되며, 모두 Docker 컨테이너로 실행된다.

### 슬라이드 F-2: Locust 스크립트 구조 이해하기

핵심 메시지: Locust 스크립트는 4가지 핵심 요소로 구성된다.

설명 내용:
Locust 스크립트의 기본 구조:

```python
from locust import HttpUser, task, between

class MyUser(HttpUser):
    wait_time = between(1, 3)

    @task
    def my_task(self):
        self.client.get("/api/endpoint")
```

4가지 핵심 구성 요소:
- HttpUser: 가상 사용자(VU)를 정의하는 클래스. 이 클래스를 상속받아 테스트 시나리오를 작성한다.
- wait_time = between(1, 3): 요청 간 대기 시간(Think Time). 1초에서 3초 사이의 랜덤 값으로 설정된다.
- @task: 가상 사용자가 반복 수행할 작업을 정의하는 데코레이터. 여러 개의 @task를 정의할 수 있다.
- self.client: HTTP 요청을 보내는 클라이언트 객체. get, post, put, delete 등의 메서드를 제공한다.

발표자 노트: 이 슬라이드에서 각 요소를 하나씩 짚어주며 설명한다. 수강생들이 코드를 따라 타이핑할 시간을 준다.

디자인 지시: 코드 블록과 함께 각 요소를 화살표로 가리키며 설명하는 주석 형태의 레이아웃.

### 슬라이드 F-3: 실습 따라하기 — 스크립트 작성

핵심 메시지: locustfile.py 파일을 만들고 기본 부하 테스트 스크립트를 작성한다.

설명 내용:
실습 절차:

Step 1. VS Code에서 locustfile.py 파일을 새로 생성한다.

Step 2. 아래 코드를 입력한다.

```python
from locust import HttpUser, task, between

class BasicUser(HttpUser):
    """기본 부하 테스트 시나리오"""
    wait_time = between(1, 3)

    @task(3)
    def get_items(self):
        """상품 목록 조회 — 가중치 3"""
        self.client.get("/api/items", name="상품 목록")

    @task(1)
    def get_item_detail(self):
        """상품 상세 조회 — 가중치 1"""
        self.client.get("/api/items/1", name="상품 상세")
```

코드 설명:
- @task(3)과 @task(1): 가중치 설정. 상품 목록 조회가 상세 조회보다 3배 더 자주 실행된다. 실제 사용 패턴에서 "목록 조회가 상세 조회보다 빈번한" 비율을 반영할 때 유용하다.
- name 파라미터: Locust 결과에서 이 이름으로 요청을 구분하여 보여준다.

발표자 노트: 수강생들이 코드를 직접 타이핑하도록 유도한다. 타이핑이 끝난 사람은 손을 들어 확인한다.

디자인 지시: 코드 블록을 크게 보여주고, 핵심 부분에 하이라이트 표시.

### 슬라이드 G-2: k6 스크립트 구조 이해하기

핵심 메시지: k6 스크립트는 options(설정)과 default function(시나리오)으로 구성된다.

설명 내용:
k6 스크립트의 기본 구조:

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  vus: 10,
  duration: '30s',
};

export default function () {
  const res = http.get('http://target-server:8080/');
  check(res, {
    '상태코드 200': (r) => r.status === 200,
    '응답시간 500ms 이하': (r) => r.timings.duration < 500,
  });
  sleep(1);
}
```

4가지 핵심 구성 요소:
- options: 테스트 설정. VU 수, 지속 시간, Stages, Thresholds 등을 정의한다.
- export default function: 각 VU가 반복 실행하는 시나리오 함수. Locust의 @task에 해당한다.
- check(): 응답 값을 검증하는 함수. HTTP 상태 코드, 응답 시간, 응답 본문 등을 검증할 수 있다. Pass/Fail이 기록된다.
- sleep(): Think Time을 설정하는 함수. 초 단위로 대기한다.

발표자 노트: Locust 스크립트와 대비하여 설명하면 이해가 빠르다. "Locust의 HttpUser가 k6에서는 default function입니다."

디자인 지시: 코드 블록과 함께 각 요소를 화살표로 가리키며 설명하는 주석 형태의 레이아웃.

### 슬라이드 G-3: 실습 따라하기 — Stages와 Thresholds 스크립트 작성

핵심 메시지: Stages로 부하 패턴을 설계하고, Thresholds로 성공 기준을 정의한다.

설명 내용:
실습 절차:

Step 1. VS Code에서 script.js 파일을 새로 생성한다.

Step 2. 아래 코드를 입력한다.

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '30s', target: 5 },   // 30초간 0→5명 증가 (Ramp-up)
    { duration: '1m', target: 5 },    // 1분간 5명 유지 (Steady)
    { duration: '30s', target: 0 },   // 30초간 5→0명 감소 (Ramp-down)
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],  // p95 응답 시간 500ms 이하
    http_req_failed: ['rate<0.01'],    // 에러율 1% 미만
  },
};

export default function () {
  const res = http.get('http://host.docker.internal:8080/api/items');
  check(res, {
    '상태코드 200': (r) => r.status === 200,
  });
  sleep(Math.random() * 2 + 1);  // 1~3초 랜덤 대기
}
```

코드 설명:
- stages: 이론에서 배운 Ramp-up → Steady → Ramp-down을 코드로 정의한 것이다. 30초간 사용자를 0명에서 5명으로 늘리고, 1분간 5명을 유지하고, 30초간 다시 0명으로 줄인다.
- thresholds: 성공 기준을 코드로 정의한다. p95 응답 시간이 500ms를 넘으면 테스트 실패, 에러율이 1%를 넘으면 테스트 실패로 판정한다.
- sleep(Math.random() * 2 + 1): 1초에서 3초 사이의 랜덤한 Think Time. Locust의 between(1, 3)과 동일한 역할이다.

발표자 노트: stages의 각 단계와 이론에서 배운 Ramp-up/Steady/Ramp-down 개념을 연결하여 설명한다.

디자인 지시: 코드 블록을 크게 보여주고, stages와 thresholds 부분에 하이라이트 표시.

### 슬라이드 G-4: 실습 따라하기 — k6 실행 및 결과 확인

핵심 메시지: k6를 실행하고 터미널 출력에서 결과를 확인한다.

설명 내용:
실습 절차:

Step 3. 터미널에서 k6 폴더로 이동한 뒤 아래 명령어를 실행한다.

Step 4. 테스트가 완료되면 터미널에 출력되는 결과를 확인한다.

결과 출력 예시:
- checks: 100.00% ✓ 450 ✗ 0 → 모든 응답 검증 통과
- http_req_duration: avg=135ms, p(90)=250ms, p(95)=380ms → 평균 135ms, 95% 요청이 380ms 이내
- http_req_failed: 0.00% ✓ 0 ✗ 600 → 에러 없음
- http_reqs: 600, 3.33/s → 초당 약 3.3건 처리(RPS)

Thresholds 결과:
- http_req_duration p(95)<500 → 실제 380ms → pass ✓
- http_req_failed rate<0.01 → 실제 0.00% → pass ✓

모든 Thresholds가 통과하면 종료 코드 0(성공)이 반환된다. 하나라도 실패하면 종료 코드 99(실패)가 반환된다. CI/CD 파이프라인에서 이 종료 코드로 자동 합격/불합격 판정이 가능하다.

발표자 노트: Locust에서는 사람이 직접 결과를 보고 판단했지만, k6에서는 Thresholds가 자동으로 판단해 준다는 점을 강조.

디자인 지시: 터미널 출력 화면을 스크린샷 형태로 보여주고, 핵심 수치에 하이라이트 표시. pass/fail 표시를 크게 강조.

### 슬라이드 G-5: 실습 따라하기 — 시나리오 기반 테스트 (group 활용)

핵심 메시지: group()으로 사용자 행동 흐름을 단계별로 나누어 테스트한다.

설명 내용:
실습 절차:

Step 5. script.js 파일의 default function 부분을 아래와 같이 수정한다.

```javascript
import http from 'k6/http';
import { check, group, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '30s', target: 5 },
    { duration: '1m', target: 5 },
    { duration: '30s', target: 0 },
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],
    http_req_failed: ['rate<0.01'],
  },
};

export default function () {
  group('01_목록조회', function () {
    const listRes = http.get('http://host.docker.internal:8080/api/items');
    check(listRes, {
      '목록 조회 성공': (r) => r.status === 200,
    });
    sleep(2);
  });

  group('02_상세조회', function () {
    const detailRes = http.get('http://host.docker.internal:8080/api/items/1');
    check(detailRes, { '상세 조회 성공': (r) => r.status === 200 });
    sleep(1);
  });

  group('03_상품생성', function () {
    const createRes = http.post(
      'http://host.docker.internal:8080/api/items',
      JSON.stringify({ name: '테스트 상품', value: 100 }),
      { headers: { 'Content-Type': 'application/json' } }
    );
    check(createRes, { '상품 생성 성공': (r) => r.status === 201 });
    sleep(1);
  });
}
```

코드 설명:
- group('이름', function() {...}): 시나리오의 각 단계를 이름을 붙여 구분한다.
- 결과에서 각 group별로 성능 지표가 분리되어 표시된다. 이를 통해 "목록 조회에서 병목이 발생하는지", "상품 생성에서 느린지"를 한눈에 파악할 수 있다.
- Locust의 SequentialTaskSet과 유사한 개념이지만, k6의 group()은 더 유연하게 조합할 수 있다.

Step 6. 다시 k6를 실행하여 결과를 확인한다.

발표자 노트: 시나리오 기반 테스트가 왜 중요한지를 사례 2(쇼핑몰)와 연결하여 설명한다. "아까 배운 블랙프라이데이 사례에서, 조회는 괜찮았지만 결제에서 문제가 생겼죠? group으로 나누면 이런 병목을 바로 찾을 수 있습니다."

디자인 지시: 코드 블록과 함께, 사용자 행동 흐름(목록조회→상세조회→상품생성)을 플로우차트로 표현.

### 슬라이드 G-6: 실습 따라하기 — 시나리오 직접 수정해보기

핵심 메시지: stages, thresholds, 테스트 대상을 직접 바꿔 보면서 결과가 어떻게 달라지는지 체험한다.

설명 내용:
앞에서 작성한 script.js를 아래 3가지 미션에 따라 수정하고, 매번 다시 실행하여 결과 변화를 관찰한다.

미션 1 — 스파이크 테스트로 바꿔보기:
stages를 아래와 같이 수정하면 부하 테스트가 스파이크 테스트로 바뀐다.

```javascript
stages: [
  { duration: '10s', target: 2 },   // 10초간 소수 사용자
  { duration: '5s',  target: 30 },  // 5초 만에 30명으로 급증 (스파이크!)
  { duration: '30s', target: 30 },  // 30초간 유지하며 서버 반응 관찰
  { duration: '10s', target: 0 },   // 정리
],
```

관찰 포인트: 급증 구간에서 응답 시간과 에러율이 어떻게 변하는가? 서버가 스파이크를 버텨내는가?

미션 2 — 성능 기준을 더 엄격하게 바꿔보기:
thresholds를 아래와 같이 수정하면 더 까다로운 기준으로 합격/불합격을 판정한다.

```javascript
thresholds: {
  http_req_duration: ['p(95)<200'],   // 기존 500ms → 200ms로 강화
  http_req_failed: ['rate<0.001'],    // 기존 1% → 0.1%로 강화
},
```

관찰 포인트: 기존에 pass였던 테스트가 fail로 바뀌는가? Thresholds를 어떤 기준으로 설정해야 하는지 생각해 본다.

미션 3 — 느린 API를 테스트 대상에 추가하기:
대상 서버에는 의도적으로 지연이 발생하는 API가 있다. group을 하나 더 추가하여 병목을 직접 관찰한다.

```javascript
group('04_지연API', function () {
  const slowRes = http.get('http://host.docker.internal:8080/api/slow');
  check(slowRes, {
    '지연 API 응답 성공': (r) => r.status === 200,
    '응답시간 1초 이내': (r) => r.timings.duration < 1000,
  });
  sleep(1);
});
```

관찰 포인트: 전체 시나리오에 느린 API가 포함되면 전체 응답 시간과 RPS가 어떻게 변하는가? group별 결과를 비교하면 병목이 어디인지 바로 보인다.

실습 정리:

| 수정 항목 | 변경 전 | 변경 후 | 확인할 것 |
|-----------|---------|---------|-----------|
| stages | 점진적 증가(Ramp-up) | 급격한 스파이크 | 응답 시간 급증, 에러 발생 여부 |
| thresholds | p(95)<500ms | p(95)<200ms | pass → fail 변화 |
| 테스트 대상 | /api/items 중심 | /api/slow 추가 | group별 병목 비교 |

발표자 노트: 미션은 시간에 따라 1~2개만 선택하여 진행해도 된다. 핵심은 "스크립트의 어떤 부분을 바꾸면 어떤 결과가 달라지는지"를 직접 체험하는 것이다. 미션 1(스파이크)은 이론에서 배운 테스트 유형과 바로 연결되므로 가장 효과적이다.

디자인 지시: 3개 미션을 카드 형태로 나란히 배치. 각 카드에 수정할 코드 스니펫과 관찰 포인트를 포함. "직접 해보세요!" 느낌의 인터랙티브한 디자인.

### 슬라이드 G-7: k6 결과 읽기 실습

핵심 메시지: k6 터미널 결과의 각 지표가 무엇을 의미하는지 함께 해석한다.

설명 내용:
터미널 결과에서 확인해야 할 핵심 지표:

- checks: 100.00%이면 모든 응답 검증 통과. 0%에 가까울수록 문제가 있다.
- http_req_duration avg: 전체 요청의 평균 응답 시간. 전반적인 서버 성능을 보여준다.
- http_req_duration p(95): 95% 요청이 이 시간 이내에 완료. 대부분의 사용자 경험을 대표한다.
- http_req_failed: 에러율. 0%에 가까울수록 안정적이다.
- http_reqs: 초당 처리한 요청 수(RPS). 서버의 처리량을 나타낸다.
- iterations: 초당 완료한 시나리오 수. group이 포함된 전체 시나리오 1회 완료를 1 iteration으로 카운트한다.
- vus: 활성 가상 사용자 수. stages에 따라 변화한다.

Thresholds 결과 읽기:
- ✓ 표시 = 기준 통과 (pass)
- ✗ 표시 = 기준 미달 (fail)

디자인 지시: 터미널 결과를 크게 보여주고, 각 지표에 번호를 매겨 설명과 연결.

핵심 포인트: 숫자는 "결과"를 알려주고, 그래프는 "추세"를 알려준다. 성능 테스트에서는 둘 다 필요하다.

디자인 지시: 왼쪽에 숫자 테이블, 오른쪽에 같은 데이터의 그래프를 나란히 배치. 그래프에서 발견되는 패턴을 원으로 강조.

- 1초에서 5초로 늘어나면 이탈률이 90% 증가한다.
- 1초에서 6초로 늘어나면 이탈률이 106% 증가한다.
- 1초에서 10초로 늘어나면 이탈률이 123% 증가한다.
출처: Google/SOASTA Research, 2017

Amazon의 연구 결과:
- 페이지 로딩 시간이 100ms 증가할 때마다 매출이 1% 감소한다.
출처: Amazon Internal Study (Greg Linden, 2006)

Akamai의 연구 결과:
- 웹사이트 로딩이 2초를 넘기면 사용자의 47%가 이탈한다.
- 모바일 사용자의 53%가 3초 이상 로딩되는 사이트를 이탈한다.
출처: Akamai Online Retail Performance Report

## 업계 표준 성능 기준

일반적인 웹 애플리케이션 성능 기준:
- 응답 시간: p95 기준 500ms 이하를 양호, 1초 이하를 허용 가능, 3초 이상을 불량으로 판단하는 경우가 많다.
- 에러율: 1% 이하를 안정적으로 판단한다. 0.1% 이하가 이상적이다.
- 가용성(Availability): 99.9% 이상을 목표로 하는 경우가 많다. 99.9%는 연간 약 8.76시간의 다운타임을 의미한다. 99.99%는 연간 약 52.6분의 다운타임을 의미한다.

SLA(Service Level Agreement) 관련:
- SLA는 서비스 제공자가 고객에게 약속하는 서비스 수준이다.
- SLO(Service Level Objective)는 SLA를 달성하기 위한 내부 목표이다.
- SLI(Service Level Indicator)는 SLO를 측정하기 위한 구체적인 지표이다.
- 성능 테스트의 Thresholds는 SLO를 검증하는 수단으로 활용된다.

## 성능 테스트 도구 생태계

주요 성능 테스트 도구 비교:
- Apache JMeter: Java 기반의 가장 역사가 오래된 성능 테스트 도구. GUI 기반으로 시나리오 작성이 가능하지만, 대규모 테스트 시 리소스 소비가 크다.
- Gatling: Scala 기반의 성능 테스트 도구. 코드 기반 시나리오 작성과 상세한 HTML 리포트가 강점이다.
- Locust: Python 기반. 코드 기반 시나리오와 Web UI가 강점. 분산 테스트를 지원한다.
- k6: JavaScript 기반. CLI 중심으로 CI/CD 연동에 최적화되어 있다. Grafana Labs에서 관리한다.
- Artillery: Node.js 기반. YAML로 시나리오를 정의할 수 있어 간편하다.
- nGrinder: Naver에서 개발한 Java 기반 성능 테스트 도구. 한국 기업에서 많이 사용된다.

이 강의에서 Locust와 k6를 선택한 이유:
1. 코드 기반으로 시나리오를 작성하여 버전 관리가 가능하다.
2. QA 엔지니어에게 친숙한 Python과 JavaScript를 사용한다.
3. Docker를 통한 설치가 간편하고 별도의 라이선스가 필요 없다.
4. 두 도구의 접근 방식이 다르므로 비교하며 학습 효과를 높일 수 있다.

## HTTP 상태 코드와 성능 테스트

성능 테스트에서 자주 만나는 HTTP 상태 코드:
- 200 OK: 정상 응답
- 201 Created: 리소스 생성 성공
- 400 Bad Request: 잘못된 요청. 테스트 데이터가 올바르지 않을 때 발생할 수 있다.
- 401 Unauthorized: 인증 실패. 토큰이 만료되었거나 로그인 시나리오에 문제가 있을 때.
- 403 Forbidden: 접근 권한 없음.
- 404 Not Found: 요청한 리소스를 찾을 수 없음. 테스트 URL이 잘못되었을 때.
- 429 Too Many Requests: 요청 빈도 제한(Rate Limiting)에 걸렸을 때. 성능 테스트에서 자주 발생한다.
- 500 Internal Server Error: 서버 내부 오류. 부하가 증가하면서 서버가 처리하지 못할 때 발생.
- 502 Bad Gateway: 게이트웨이/프록시 서버가 백엔드 서버로부터 잘못된 응답을 받았을 때.
- 503 Service Unavailable: 서버가 과부하 상태이거나 유지보수 중일 때. 성능 테스트에서 서버 한계에 도달하면 발생.
- 504 Gateway Timeout: 게이트웨이/프록시 서버가 백엔드 서버로부터 응답을 시간 내에 받지 못했을 때.

## 성능 병목의 일반적인 원인

서버 측 병목:
- CPU 병목: 연산이 많은 요청에서 CPU 사용률이 100%에 도달
- 메모리 병목: 메모리 부족으로 스왑(Swap) 발생, GC(Garbage Collection) 과다
- DB 병목: 느린 쿼리, 인덱스 미설정, 커넥션 풀 고갈
- 네트워크 병목: 대역폭 한계, 높은 레이턴시
- I/O 병목: 디스크 읽기/쓰기 속도 한계

애플리케이션 측 병목:
- N+1 쿼리 문제: 반복문 안에서 DB 쿼리를 호출하여 요청 수가 기하급수적으로 증가
- 동기식 처리: 비동기 처리가 가능한 작업을 동기로 처리하여 쓰레드가 블로킹
- 메모리 누수: 오래 실행할수록 메모리 사용량이 증가하여 성능 저하
- 캐시 미활용: 동일한 데이터를 매번 DB에서 조회

## Little's Law — 성능 테스트의 수학적 기반

Little's Law (리틀의 법칙):
L = λ × W

여기서:
- L = 시스템 내 평균 요청 수 (동시 사용자 수)
- λ = 도착률 (초당 요청 수, RPS와 유사)
- W = 평균 응답 시간 (초)

예시:
- 동시 사용자 100명, 평균 응답 시간 2초인 경우
- RPS = 100 / 2 = 50 RPS
- 즉, 서버가 초당 50건의 요청을 처리해야 100명의 동시 사용자를 지원할 수 있다.

이 법칙을 활용하면:
- 목표 동시 사용자 수와 응답 시간으로부터 필요한 RPS를 계산할 수 있다.
- 현재 서버의 RPS와 응답 시간으로부터 지원 가능한 동시 사용자 수를 추정할 수 있다.

## API란? — 성능 테스트의 대상 이해하기

성능 테스트를 시작하기 전에, 우리가 정확히 무엇을 테스트하는지 이해해야 한다. 성능 테스트의 대상은 대부분 **API(Application Programming Interface)**이다.

API의 정의:
- API는 소프트웨어 간의 통신 규약이다. 한 프로그램이 다른 프로그램에 요청을 보내고 응답을 받기 위한 약속된 방법을 정의한다.

비유 — 식당의 메뉴판과 주문 시스템:
- 고객(클라이언트)은 메뉴판(API 명세)을 보고 원하는 요리를 주문한다.
- 웨이터(API)가 주문을 주방(서버)에 전달한다.
- 주방(서버)이 요리를 만들어 웨이터를 통해 고객에게 전달한다.
- 고객은 주방 내부가 어떻게 동작하는지 알 필요가 없다. 메뉴판에 적힌 규칙(API)만 알면 된다.

REST API 기본 개념:
- REST(Representational State Transfer)는 가장 널리 사용되는 API 설계 방식이다.
- HTTP 메서드로 수행할 작업의 종류를 구분한다:
  - GET: 데이터 조회 — 예: `GET /api/items` (상품 목록 조회)
  - POST: 데이터 생성 — 예: `POST /api/items` (새 상품 등록)
  - PUT: 데이터 수정 — 예: `PUT /api/items/1` (1번 상품 수정)
  - DELETE: 데이터 삭제 — 예: `DELETE /api/items/1` (1번 상품 삭제)
- 엔드포인트(Endpoint)는 API에 접근하기 위한 URL 경로이다. 예: `https://api.example.com/api/items`

요청(Request)과 응답(Response)의 구조:
- 요청: HTTP 메서드 + URL + 헤더(Header) + 본문(Body)
  - 헤더: 인증 토큰, 콘텐츠 타입 등 부가 정보를 담는다. 예: `Content-Type: application/json`
  - 본문: POST, PUT 요청 시 전송할 데이터를 담는다. 예: `{"name": "새 상품", "price": 10000}`
- 응답: 상태 코드 + 헤더 + 본문
  - 상태 코드: 요청 처리 결과를 숫자로 표현한다. 예: 200(성공), 404(없음), 500(서버 오류)
  - 본문: 서버가 반환하는 데이터를 담는다. 예: `{"id": 1, "name": "상품A", "price": 10000}`

API와 웹 페이지의 차이:
- 웹 페이지: 사람이 브라우저로 보는 HTML 문서이다. 텍스트, 이미지, 스타일이 포함되어 있다.
- API: 프로그램이 주고받는 구조화된 데이터이다. 주로 JSON(JavaScript Object Notation) 형식을 사용한다.
- 성능 테스트에서는 브라우저를 띄우지 않고, API에 직접 HTTP 요청을 보내 서버의 순수한 처리 능력을 측정한다.

성능 테스트와의 관계:
- 성능 테스트는 API에 대량의 요청을 동시에 보내 서버가 얼마나 많은 요청을 얼마나 빠르게 처리할 수 있는지 측정하는 것이다.
- 이 강의 실습에서 테스트하는 API 엔드포인트:
  - `GET /api/items` — 상품 목록 조회
  - `GET /api/items/{id}` — 특정 상품 상세 조회
  - `POST /api/items` — 새 상품 등록
- API 설계가 성능에 미치는 영향:
  - 페이지네이션(Pagination): 전체 데이터를 한 번에 반환하면 응답 크기가 커져 느려진다. 페이지 단위로 나누어 반환하면 성능이 개선된다.
  - 캐싱(Caching): 자주 요청되는 데이터를 메모리에 저장해 두면 DB 조회 없이 빠르게 응답할 수 있다.
  - Rate Limiting: 클라이언트별 요청 횟수를 제한하여 서버를 보호한다. 성능 테스트 시 429(Too Many Requests) 응답이 발생하면 Rate Limiting이 적용된 것이다.

## Docker란? — 성능 테스트 환경을 쉽게 구성하는 방법

이 강의의 모든 실습은 Docker를 사용하여 환경을 구성한다. Docker가 무엇이고, 왜 성능 테스트에서 활용하는지 알아보자.

Docker의 정의:
- Docker는 애플리케이션을 **컨테이너(Container)**라는 단위로 패키징하여 어디서든 동일하게 실행할 수 있게 해주는 기술이다.

비유 — 이삿짐 컨테이너:
- 이사할 때 가구와 살림살이를 컨테이너에 넣으면, 어떤 트럭에 실어도 내용물은 동일하다.
- Docker 컨테이너도 마찬가지로, 애플리케이션과 실행에 필요한 모든 것(코드, 라이브러리, 설정)을 하나의 패키지로 묶어서 어떤 컴퓨터에서든 동일하게 실행된다.

핵심 개념:
- 이미지(Image): 실행에 필요한 모든 것이 담긴 패키지이다. 건물의 설계도에 해당한다. Docker Hub 등에서 다운로드하거나 직접 만들 수 있다.
- 컨테이너(Container): 이미지를 실행한 인스턴스이다. 설계도로 지은 실제 건물에 해당한다. 하나의 이미지로 여러 컨테이너를 만들 수 있다.
- Docker Compose: 여러 컨테이너를 한 번에 정의하고 실행하는 도구이다. `docker-compose.yml` 파일에 필요한 서비스들을 선언하면 `docker compose up` 명령 한 번으로 전체 환경을 구동할 수 있다.

가상 머신(VM)과의 비교:
- VM(가상 머신): 하드웨어 위에 게스트 OS 전체를 설치하여 실행한다. 용량이 크고(수 GB) 시작하는 데 수 분이 걸린다.
- Container(컨테이너): 호스트 OS의 커널을 공유하고 애플리케이션만 격리하여 실행한다. 용량이 작고(수 MB~수백 MB) 시작하는 데 수 초면 충분하다.
- 성능 테스트 관점에서 컨테이너는 가벼워서 여러 구성 요소(테스트 도구, 모니터링 시스템, 대상 서버)를 한 대의 노트북에서도 실행할 수 있다.

이 강의에서 Docker를 사용하는 이유:
- 동일한 환경: 모든 수강생이 OS나 설치 환경에 관계없이 동일한 조건에서 실습할 수 있다. "제 컴퓨터에서는 안 돼요" 문제를 제거한다.
- 간편한 설치: 복잡한 설치 과정 없이 `docker compose up` 한 번으로 대상 서버, 테스트 도구, 모니터링 시스템을 모두 구동한다.
- 깨끗한 정리: 실습 후 `docker compose down`으로 모든 컨테이너를 종료하고 리소스를 반환한다. 실습이 로컬 환경을 오염시키지 않는다.

이 강의의 Docker 구성:
- `target-server/`: 성능 테스트의 대상이 되는 FastAPI 기반 웹 서버이다.
- `locust/`: Locust master + worker 구조로 구성된다. master는 Web UI를 제공하고 worker가 실제 부하를 생성한다.
- `k6/`: k6 테스트 러너 + InfluxDB(결과 저장) + Grafana(결과 시각화)로 구성된다.

`host.docker.internal`이란?:
- 컨테이너는 격리된 네트워크 환경에서 실행되므로 호스트 머신(내 컴퓨터)에 직접 접근할 수 없다.
- `host.docker.internal`은 컨테이너 내부에서 호스트 머신의 IP 주소를 가리키는 특수 DNS 이름이다.
- 예: 대상 서버가 호스트의 8000번 포트에서 실행 중이라면, 컨테이너 안에서 `http://host.docker.internal:8000`으로 접근할 수 있다.
- 이 강의의 Locust와 k6 스크립트에서 대상 서버 주소로 `host.docker.internal`을 사용하는 이유가 바로 이것이다.

## MSA(Microservices Architecture)란? — 현대 서비스 구조와 성능 테스트

실무에서 성능 테스트를 수행하면, 단순히 하나의 서버를 테스트하는 것이 아니라 여러 서비스가 연결된 복잡한 환경을 다루게 된다. 이 섹션에서는 현대 서비스 아키텍처의 기본 개념과 성능 테스트에 미치는 영향을 설명한다.

모놀리식(Monolithic) vs MSA 비교:
- 모놀리식 아키텍처: 하나의 큰 애플리케이션에 모든 기능(사용자 관리, 상품 관리, 주문 처리, 결제 등)이 포함된 구조이다.
  - 비유: 하나의 건물에 모든 부서(인사, 영업, 개발, 재무)가 같이 있는 회사.
  - 장점: 구조가 단순하고, 서비스 간 통신 비용이 없다. 하나의 서버만 테스트하면 된다.
  - 단점: 기능 하나를 수정하려면 전체를 다시 배포해야 한다. 규모가 커지면 유지보수가 어려워진다.
- MSA(Microservices Architecture): 기능별로 독립된 작은 서비스로 분리한 구조이다.
  - 비유: 각 부서(인사, 영업, 개발, 재무)가 별도의 건물에 위치하고, 전화나 메신저로 소통하는 회사.
  - 장점: 각 서비스를 독립적으로 배포하고 확장할 수 있다. 팀별로 다른 기술 스택을 사용할 수 있다.
  - 단점: 서비스 간 네트워크 통신이 필요하여 복잡도가 증가한다.

MSA의 핵심 특징:
- 서비스별 독립 배포: 상품 서비스만 업데이트해도 다른 서비스에 영향을 주지 않는다.
- 서비스별 독립 확장: 트래픽이 많은 서비스만 서버를 추가로 늘릴 수 있다.
- 서비스 간 통신 방식:
  - REST API: HTTP를 사용한 동기 통신. 가장 보편적인 방식이다.
  - gRPC: Google이 개발한 고성능 통신 프로토콜. REST보다 빠르고 효율적이다.
  - 메시지 큐(Message Queue): Kafka, RabbitMQ 등을 사용한 비동기 통신. 서비스 간 느슨한 결합을 가능하게 한다.
- 각 서비스가 자체 데이터베이스를 가질 수 있다. 사용자 서비스는 MySQL, 상품 서비스는 MongoDB, 검색 서비스는 Elasticsearch를 사용하는 식이다.

MSA 환경에서 성능 테스트가 중요한 이유:
- 하나의 사용자 요청이 내부적으로 여러 서비스를 거쳐 처리된다:
  - 예: "상품 주문" 요청 → 사용자 서비스(인증 확인) → 상품 서비스(상품 정보 조회) → 재고 서비스(재고 확인 및 차감) → 결제 서비스(결제 처리) → 알림 서비스(주문 확인 알림 발송)
  - 사용자는 하나의 "주문" 버튼을 눌렀을 뿐이지만, 내부적으로 5개 이상의 서비스가 순차적으로 호출된다.
- 서비스 간 네트워크 호출이 추가되므로 레이턴시(지연)가 누적된다. 각 서비스의 응답 시간이 50ms라 해도 5개 서비스를 거치면 최소 250ms가 소요된다.
- 연쇄 장애(Cascading Failure): 한 서비스의 장애가 다른 서비스로 전파될 수 있다. 결제 서비스가 응답하지 않으면 주문 서비스도 대기하게 되고, 결국 전체 시스템이 마비될 수 있다.

MSA 환경의 성능 테스트 고려사항:
- End-to-End 시나리오 테스트: 개별 서비스가 아닌 사용자 관점의 전체 흐름을 테스트해야 한다. 이 강의의 섹션 A-3에서 다룬 쇼핑몰 사례(상품 조회 → 장바구니 → 결제)가 이에 해당한다.
- 병목 서비스 식별: 서비스별 성능 한계가 다를 수 있다. 전체 응답 시간은 가장 느린 서비스에 의해 결정된다. 체인에서 가장 약한 고리가 전체 강도를 결정하는 것과 같다.
- Circuit Breaker(회로 차단기): 장애가 발생한 서비스로의 호출을 자동으로 차단하여 전체 시스템을 보호하는 패턴이다. 가정의 차단기가 과부하 시 전기를 끊어 화재를 방지하는 것과 같은 원리이다.
- 분산 추적(Distributed Tracing): Jaeger, Zipkin 등의 도구를 사용하여 하나의 요청이 여러 서비스를 거치는 전체 흐름을 추적할 수 있다. 어느 서비스에서 시간이 오래 걸리는지 파악하는 데 필수적이다.

## Kubernetes(k8s)에서의 성능 테스트 — 클라우드 환경의 확장

실무에서 대부분의 서비스는 클라우드 환경에 배포되며, Kubernetes(k8s)가 컨테이너 관리의 표준으로 자리잡았다. 이 섹션에서는 Kubernetes 환경에서 성능 테스트가 어떻게 달라지는지 소개한다.

Kubernetes의 정의:
- Kubernetes(k8s)는 컨테이너를 자동으로 배포, 확장, 관리하는 **오케스트레이션(Orchestration)** 플랫폼이다.
- 비유: Docker가 "컨테이너 하나"를 만들고 실행하는 것이라면, Kubernetes는 "컨테이너 수백 개의 함대"를 지휘하는 사령관이다. 어떤 컨테이너를 어느 서버에 배치할지, 장애가 나면 어떻게 대체할지, 부하가 늘면 얼마나 확장할지를 자동으로 결정한다.

핵심 개념 (성능 테스트 관점에서):
- Pod: 컨테이너가 실행되는 최소 단위이다. 보통 하나의 Pod에 하나의 컨테이너가 실행된다. 동일한 Pod를 여러 개 만들어 부하를 분산한다.
- Service: Pod에 접근하기 위한 네트워크 엔드포인트이다. 여러 Pod 앞에 로드밸런서 역할을 하여 요청을 분배한다.
- HPA(Horizontal Pod Autoscaler): 부하에 따라 Pod 수를 자동으로 조절하는 기능이다.
  - 예: CPU 사용률이 70%를 넘으면 Pod를 2개에서 4개로 자동 증가시킨다.
  - 예: 부하가 줄어들면 Pod 수를 다시 2개로 축소한다.
- 노드(Node): Pod가 실행되는 물리 서버 또는 가상 서버이다. 하나의 노드에 여러 Pod가 배치될 수 있다.

k8s 환경에서의 성능 테스트 특성:
- Auto-scaling의 영향:
  - 부하가 증가하면 Pod가 자동으로 늘어나므로, 고정된 인프라에서의 테스트와 결과가 다르다.
  - 같은 테스트를 반복하면 매번 결과가 달라질 수 있다. 인프라가 동적으로 변하기 때문이다.
  - 성능 테스트 시 Auto-scaling을 비활성화하고 고정 Pod 수로 테스트하는 것이 기준선(Baseline)을 잡는 데 유리하다.
- 스케일링 지연:
  - Pod 생성에는 시간이 걸린다. 새로운 컨테이너 이미지를 다운로드하고, 애플리케이션이 시작되고, 헬스 체크를 통과해야 한다.
  - 스파이크 트래픽(갑자기 급증하는 트래픽)이 발생하면 Pod가 충분히 늘어나기 전에 기존 Pod가 과부하에 빠질 수 있다.
  - 이것이 스파이크 테스트가 중요한 이유이다. 자동 확장이 있더라도 확장 속도보다 트래픽 증가가 빠르면 장애가 발생한다.
- 리소스 제한:
  - k8s에서는 Pod별로 CPU와 메모리 사용량에 제한(Limit)을 설정한다.
  - CPU 제한을 초과하면 스로틀링(Throttling)이 발생하여 처리 속도가 강제로 낮아진다.
  - 메모리 제한을 초과하면 OOM Kill(Out of Memory Kill)로 Pod가 강제 종료된다.
  - 성능 테스트 시 Pod의 리소스 사용량을 모니터링하면 제한이 적절한지 확인할 수 있다.
- 네트워크 고려사항:
  - 서비스 메시(Service Mesh): Istio, Linkerd 같은 서비스 메시를 사용하면 서비스 간 통신에 보안, 관찰 기능이 추가되지만 약간의 레이턴시가 추가된다.
  - 이 추가 레이턴시가 성능 테스트 결과에 반영되므로, 서비스 메시 유무에 따라 결과가 달라질 수 있다.

k8s에서 k6 활용:
- k6-operator: Kubernetes에서 k6 테스트를 Job으로 실행할 수 있는 오퍼레이터이다. 여러 Pod에서 k6를 동시에 실행하여 대규모 분산 부하 테스트를 수행할 수 있다.
- 모니터링 통합: Prometheus로 인프라 메트릭(CPU, 메모리, Pod 수)을 수집하고 Grafana로 시각화하면, k6 성능 테스트 결과와 인프라 상태를 하나의 대시보드에서 함께 관찰할 수 있다.

이 강의와의 관계:
- 이 강의에서는 로컬 Docker 환경에서 성능 테스트의 기초를 다룬다. 개념과 도구 사용법을 익히는 데 집중한다.
- 실무에서는 동일한 k6 스크립트를 스테이징(Staging) 또는 프로덕션(Production) k8s 환경에서 실행하여 실제 서비스의 성능을 측정한다.
- 이 강의에서 배운 핵심 개념(Thresholds, Stages, group, 시나리오 설계)은 환경에 관계없이 동일하게 적용된다. 도구 사용법을 한번 익히면 어떤 환경에서든 활용할 수 있다.
