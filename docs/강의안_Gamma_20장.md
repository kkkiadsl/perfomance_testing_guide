## 슬라이드 3 — 성능 테스트란?

### 기능 테스트 vs 성능 테스트

| | 기능 테스트 | 성능 테스트 |
|--|--|--|
| 핵심 질문 | 올바르게 동작하는가? | 대규모 사용자가 동시에 써도 동작하는가? |
| 관심사 | 정확성 | 속도 · 안정성 · 확장성 |
| 사용자 수 | 1명 | 수십 ~ 수만 명 |

### 엘리베이터 비유

- **기능 테스트** → 버튼을 누르면 해당 층으로 이동하는가?
- **성능 테스트** → 정원 10명인 엘리베이터에 15명이 타면 어떻게 되는가?

> 기능이 "정상"이어도, 부하가 걸리면 전혀 다른 결과가 나올 수 있다.

## 슬라이드 4 — 성능 장애는 실제로 일어난다

### 사례 1 — "새벽에 열었는데 서버가 터졌어요"

인기 콘서트 티켓 오픈 → 수십만 명 동시 접속

- 페이지 로딩 30초 이상 / 504 Gateway Timeout 대량 발생
- **원인**: 스파이크 테스트 미실시

### 사례 2 — "블랙프라이데이에 결제가 안 돼요"

평소 대비 10배 트래픽 → 결제 API 병목

- 상품 조회 200ms(정상) / 결제 API 15,000ms → Timeout / 에러율 45%
- **원인**: 결제 API를 부하 테스트 범위에서 제외

### 사례 3 — "월요일 아침마다 느려요"

전 직원 동시 접속 → DB 커넥션 풀 고갈

- 대시보드 로딩 평소 2초 → 월요일 15초
- **원인**: 피크 시간대 집중 접속 패턴 미반영

## 슬라이드 5 — QA가 성능 테스트를 알아야 하는 4가지 이유

1. **품질의 완성** — 기능이 정상이어도 느리면 사용자는 떠난다
   - Google 연구: 로딩 3초 초과 시 모바일 사용자 53% 이탈

2. **장애 예방** — 출시 전 서버 한계를 파악해야 장애를 막을 수 있다

3. **근거 기반 판단** — "느린 것 같아요" → "p95 응답시간이 2초를 초과합니다"

4. **커뮤니케이션** — "응답이 느려요" → "동시 사용자 500명 이상에서 에러율이 5%를 초과합니다"

## 슬라이드 6 — 성능 테스트 4가지 유형

| 유형 | 핵심 질문 | 부하 패턴 |
|------|-----------|-----------|
| **부하 테스트** (Load) | 평소 트래픽을 잘 처리하는가? | 일정한 수평선 |
| **스트레스 테스트** (Stress) | 어디까지 버틸 수 있는가? | 계단식 상승 |
| **스파이크 테스트** (Spike) | 갑자기 몰리면 어떻게 되는가? | 급격한 피크 |
| **내구성 테스트** (Soak) | 오래 돌려도 괜찮은가? | 장시간 수평선 |

> 용도에 맞는 테스트를 골라야 한다. 부하 테스트만 통과했다고 안심하면 안 된다.

## 슬라이드 7 — 부하 테스트 vs 스트레스 테스트

### 부하 테스트 (Load Test)

- **목적**: 예상 트래픽에서 서버가 정상 동작하는지 검증
- **언제**: 신규 서비스 오픈 전, 릴리스 전 회귀 검증
- **성공 기준**: p95 응답시간 < 500ms, 에러율 < 1%
- **핵심**: "이 선 안에서 괜찮은가?" 확인

### 스트레스 테스트 (Stress Test)

- **목적**: 서버가 버티지 못하는 지점(Breaking Point)을 탐색
- **언제**: 인프라 스펙 결정 전, 용량 계획(Capacity Planning)
- **성공 기준**: 한계점 수치 파악이 목적 (테스트가 실패해도 성공)
- **핵심**: "선이 어디까지인가?" 탐색

---

## 슬라이드 8 — 스파이크 테스트

### 스파이크 테스트가 필요한 순간

- 티켓 오픈 (0시 정각에 수십만 명 동시 접속)
- 블랙프라이데이 할인 시작 순간
- 뉴스·SNS에서 서비스가 갑자기 언급될 때

### 스파이크 테스트의 3가지 질문

1. 스파이크 발생 시 서버가 살아 있는가?
2. 스파이크가 끝난 후 서버가 정상으로 회복하는가?
3. 회복에 얼마나 걸리는가?

### 스파이크 vs 스트레스

- **스트레스**: 천천히 계단식으로 올라간다 (서버가 적응할 시간이 있다)
- **스파이크**: 갑자기 10배, 100배로 폭발한다 (서버가 준비할 틈이 없다)

---

## 슬라이드 9 — 내구성 테스트

### 내구성 테스트가 잡아내는 결함

- **메모리 누수(Memory Leak)**: 처음 100ms → 4시간 후 5,000ms
- **DB 커넥션 고갈**: 커넥션이 반환되지 않아 시간이 지나면 새 요청 처리 불가
- **파일 핸들 누수**: 열린 파일/소켓이 닫히지 않아 운영체제 자원 고갈

### 내구성 테스트의 전형적 패턴

| 시점 | 응답시간 | 에러율 | 상태 |
|------|---------|--------|------|
| 초반 1시간 | 100ms | 0% | 정상 |
| 중반 3시간 | 200ms | 0.1% | 슬금슬금 증가 신호 |
| 후반 6시간 | 3,000ms | 20% | 심각 |

## 슬라이드 10 — 응답시간과 백분위 (p95/p99)

### 왜 평균만으로는 부족한가?

요청 1~9번: 각각 100ms
요청 10번: 10,000ms (10초)

- **평균**: 1,090ms → "나쁘지 않아 보임"
- **p99**: 10,000ms → "100명 중 1명은 10초를 기다렸다!"

### 주요 측정값

| 지표 | 의미 |
|------|------|
| 평균(Average) | 전체 응답시간 합 ÷ 요청 수 |
| p50 (중앙값) | 50%의 요청이 이 시간 이내 완료 |
| p95 | 95%의 요청이 이 시간 이내 완료 |
| p99 | 99%의 요청이 이 시간 이내 완료 |

### 업계 기준

- p95 **500ms 이하** → 양호 / p95 **1,000ms 이하** → 허용 가능 / p95 **3,000ms 이상** → 불량

> Amazon 연구: 응답시간이 100ms 증가할 때마다 매출 1% 감소

## 슬라이드 11 — 처리량, 가상 사용자, Think Time

### RPS vs TPS

- **RPS (Requests Per Second)**: 초당 처리 요청 수 — 개별 HTTP 요청 기준
- **TPS (Transactions Per Second)**: 초당 처리 트랜잭션 수 — 비즈니스 단위 기준
  - 예: 로그인 → 상품 조회 → 결제 = HTTP 요청 3개 = 트랜잭션 1건

### 가상 사용자 (Virtual User, VU)

- 성능 테스트 도구가 생성하는 가상의 사용자
- VU 10명 = 10명이 동시에 같은 행동을 반복하는 것

### Think Time (대기 시간)

- 실제 사용자는 페이지를 읽거나 입력하는 데 시간을 소비한다
- Think Time 없이 테스트 → 비현실적 결과 (실제보다 훨씬 많은 요청 발생)
- Locust에서: `wait_time = between(1, 3)` → 1~3초 랜덤 대기

## 슬라이드 12 — 에러율과 테스트 3단계 구조

### 에러율 (Error Rate)

- 에러율(%) = (실패 요청 수 ÷ 전체 요청 수) × 100
- **1% 이하** → 안정적 / **5% 이상** → 사용자가 체감하는 수준

| 에러 유형 | 의미 |
|-----------|------|
| 4xx | 잘못된 요청, 권한 없음 (서버 문제 아닌 경우 多) |
| 5xx | 서버 내부 오류 — 성능 테스트의 핵심 추적 대상 |
| Timeout | 지정 시간 내 응답 없음 — 서버 과부하 신호 |

### 테스트 3단계 구조

```
[Ramp-up] → [Steady State] → [Ramp-down]
  점진적 증가     최대 부하 유지     점진적 감소
```

- **Ramp-up**: 결과 왜곡 방지, Locust에서 Spawn rate로 제어
- **Steady**: 핵심 성능 데이터를 수집하는 구간
- **Ramp-down**: 부하 해소 후 시스템이 정상 복구되는지 확인

## 슬라이드 13 — 실무에서의 성능 테스트

### 성능 테스트를 수행하는 5가지 시점

1. **신규 출시 전** — 기준선(Baseline) 수립
2. **주요 릴리스 전** — 회귀 성능 검증
3. **이벤트·프로모션 전** — 스파이크 테스트 필수
4. **인프라 변경 후** — 서버 스펙 변경, DB 마이그레이션 검증
5. **정기 회귀 테스트** — 성능 저하 조기 발견

### 5단계 성능 테스트 프로세스

```
목표 설정 → 시나리오 설계 → 환경 구성 → 실행·모니터링 → 분석·보고
```

### 흔한 5가지 실수

1. Think Time 없이 테스트 → 비현실적 결과
2. 단일 API만 테스트 → 실제 패턴과 다름
3. 로컬 결과를 절대 기준으로 삼음 → 잘못된 판단
4. Ramp-up 없이 바로 최대 부하 → 왜곡된 결과
5. 1회만 실행 후 결론 → 최소 3회 반복 필요

## 슬라이드 15 — Locust 스크립트 구조 이해하기

### 4가지 핵심 요소

```python
from locust import HttpUser, task, between

class 쇼핑몰사용자(HttpUser):
    wait_time = between(1, 3)      # ① Think Time: 1~3초 대기

    @task(3)                       # ② 가중치 3 (3배 빈번하게 실행)
    def 상품_목록_조회(self):
        self.client.get("/api/items", name="상품 목록")

    @task(1)                       # ② 가중치 1
    def 상품_상세_조회(self):
        self.client.get("/api/items/1", name="상품 상세")
```

| 요소 | 역할 |
|------|------|
| `HttpUser` | 가상 사용자 클래스 — 이것을 상속받아 시나리오 작성 |
| `wait_time` | Think Time — `between(1,3)`은 1~3초 랜덤 대기 |
| `@task(weight)` | 반복 실행 작업 — 숫자가 클수록 더 자주 실행 |
| `name` 파라미터 | 동적 URL을 하나로 묶어 집계 (`/items/1`, `/items/2` → 하나로) |

## 슬라이드 18 — 결과 해석과 보고 형식

### Statistics 탭 해석 예시

| Name | Reqs | Fails | p50 | p95 | RPS |
|------|------|-------|-----|-----|-----|
| GET /api/items | 600 | 2 | 110ms | 280ms | 3.2 |
| POST /api/items | 120 | 0 | 350ms | 680ms | 0.7 |
| Aggregated | 1,080 | 3 | 100ms | 280ms | 6.0 |

- 요청 비율 600:120 = 5:1 → @task 가중치 정확히 반영됨
- POST가 GET보다 3~4배 느림 → 쓰기 작업의 DB 부하
- 에러율 3/1,080 = **0.28%** → 1% 이하, 안정적

### 보고 형식 템플릿

| 지표 | 결과 | 기준 | 판정 |
|------|------|------|------|
| p95 응답시간 | 280ms | < 500ms | ✅ 통과 |
| 에러율 | 0.28% | < 1% | ✅ 통과 |
| 최대 안정 사용자 | 60명 | — | 참고 |

**이슈**: POST /api/items가 GET 대비 3~4배 느림. 쓰기 부하 최적화 검토 필요.

## 슬라이드 19 — 반드시 알아야 할 핵심 용어 10선

| # | 용어 | 의미 |
|---|------|------|
| 1 | VU (Virtual User) | 도구가 생성하는 가상 사용자 |
| 2 | Think Time | 요청 사이 대기 시간 — 실제 사용자 행동 반영 |
| 3 | Ramp-up | 가상 사용자를 점진적으로 증가시키는 단계 |
| 4 | RPS | 초당 처리 요청 수 — 서버의 처리 능력 |
| 5 | TPS | 초당 처리 트랜잭션 수 — 비즈니스 처리 능력 |
| 6 | p95 / p99 | 95%, 99% 백분위 응답시간 |
| 7 | 에러율 | 전체 요청 중 실패 비율 — 1% 이하가 안정적 |
| 8 | Baseline | 기준 성능 수치 — 이후 비교의 기준점 |
| 9 | Breaking Point | 에러율 급증 + 응답시간 폭증 지점 |
| 10 | Threshold | 합격/불합격 판단 기준 수치 |

### Before → After

- "서버가 느려진 것 같아요" → **"동시 사용자 60명 이상에서 p95가 1,000ms를 초과합니다"**
- "사람이 몰리면 터질 것 같아요" → **"스파이크 테스트에서 에러율이 42%까지 치솟습니다"**
